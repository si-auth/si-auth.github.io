    要約

    このガイドラインは、デジタル認証で使用するための ID 証明と登録に焦点を当てています。本人確認のプロセス中に、申請者は自分自身を確実に識別できる証拠をクレデンシャル サービス プロバイダー (CSP) に提供し、それによって CSP が有用な ID 保証レベルでその ID を主張できるようにします。このドキュメントでは、3 つの ID 保証レベルのそれぞれの技術要件を定義します。このガイドラインは、この目的外の標準の開発または使用を制限することを意図したものではありません。この出版物は、NIST Special Publication (SP) 800-63A に取って代わります。

    キーワード

    認証;クレデンシャルサービスプロバイダー。デジタル認証;身元証明;フェデレーション。

    前書き

    このドキュメントと関連する付属のボリューム ([SP800-63]、[SP800-63B]、および [SP800-63C] ) は、指定されたレベルの保証でデジタル ID を管理するためのプロセスとテクノロジーに関するガイダンスを組織に提供します。

    この文書では、個人を ID サービスに登録したり、オンライン リソースへのアクセスを提供したりする目的で、各 ID 保証レベル (IAL) で個人の ID 証明の要件を提供します。これは、ネットワーク上または対面での個人の身元証明に適用されます。

    紹介

    このセクションは有益です。

    オンラインサービスを提供する際の課題の1つは、一連のアクティビティを1人の既知の個人に関連付けることができることです。現実の主題との関連性を確実に確立することが重要な状況には、機密性の高い政府サービスへのアクセス、金融取引の実行、規制で要求される場合(金融業界の顧客識別プログラムの要件など)、リスクの高い行動に対する説明責任を確立する必要がある場合(ダムからの放水率の変更など)が含まれます。

    これらのガイドラインでは、身元証明を、オンラインサービスにアクセスする被験者と現実の人物との間にある程度の保証を得る関係を確立するプロセスとして定義しています。このドキュメントでは、「サブジェクト」および「個人」という用語は、自然人のみを指し、非個人のエンティティ、組織、または物を指すものではありません。このドキュメントは、連邦政府機関、サードパーティのクレデンシャルサービスプロバイダー(CSP)、およびID証明サービスを提供または使用するその他の組織向けのガイダンスを提供します。

    本人確認の期待される成果

    ID 証明の期待される結果は次のとおりです。

    • ID 解決: 要求された ID が、CSP またはオンライン サービスによって提供されるユーザーの母集団のコンテキスト内で、1 人の一意の個人に対応していることを判断します。
    • 証拠の検証: 提供された ID 証拠が本物で、本物で、正確であることを確認します。
    • 属性の検証: ID 証明を完了してサービスを提供するために必要な属性の最小セットであるコア属性の正確性を確認します。
    • 本人確認: 申請者が提示された証拠と属性の真の所有者であることを確認します。
    • ID 登録: ID 証明された申請者をサブスクライバーとして CSP の ID サービスに登録します。
    • 不正行為の軽減: 不正な ID を使用して、特典、サービス、データ、または資産へのアクセスを検出、対応、防止します。

    本人確認サービスには、プライバシー強化の原則 (データの最小化など) が組み込まれており、期待される結果を達成しながら申請者の負担を最小限に抑えるために、優れたユーザビリティ プラクティスが採用されることが期待されています。

    ID 保証レベル

    加入者の ID の保証 (信頼度) は、これらのガイドラインで定義されている ID 保証レベル (IAL) に関連するプロセスを使用して確立されます。連続する各 IAL は、保証を強化するために、より低い IAL の要件に基づいて構築されます。

    本人確認なし: 申請者を特定の実在の人物に関連付ける必要はありません。証拠の収集は必要ありません。属性は検証される場合と検証されない場合があります。本人確認は行われません。これにより、属性が他のビジネスプロセスの一部として検証されることが除外されるわけではありませんが、アクセスに属性の検証は必要ありません。

    IAL1: ID 証明プロセスは、要求された ID の現実世界の存在をサポートし、申請者がその ID に関連付けられていることをある程度保証します。コア属性は、身元証拠から取得されるか、申請者が自己主張します。すべてのコア属性 (セクション 2.2 を参照) は、権威あるまたは信頼できる情報源に対して検証され、属性が本人確認プロセスを受ける人物に関連付けられていることを確認するための手順が講じられます。本人確認は、CSP担当者の立ち会いの有無にかかわらず、リモートまたはオンサイトのプロセスを使用して実行されます。本人確認が正常に完了すると、申請者は加入者アカウントに登録され、加入者提供の認証者を含むすべての認証者をアカウントにバインドできます。IAL1 は、拡張性の高い攻撃 (自動登録攻撃など) を制限し、侵害された個人情報を使用した合成 ID や攻撃から保護するように設計されています。

    IAL2: IAL2 では、追加の証拠を収集し、申請者が提示された証拠の正当な所有者であることを確認するための強化されたプロセスを含む、証拠を検証し、身元を確認するためのより厳格なプロセスが必要です。IAL1と同様に、IAL2でのID証明は、CSP担当者の立ち会いの有無にかかわらず、リモートまたはオンサイトのプロセスを使用して実行できます。IAL1 で対処される脅威に加えて、IAL2 は、大規模で標的型の攻撃を制限し、基本的な証拠の改ざん、証拠の盗難、ソーシャル エンジニアリング戦術から保護するように設計されています。

    IAL3: IAL3 は、訓練を受けた CSP 担当者 (つまり、校正エージェント) が、オンサイト出席の ID 証明セッションの一環として申請者と直接対話し、少なくとも 1 つの生体認証特性を収集するための要件を追加します。オンサイトでの ID 証明セッションが成功すると、申請者が加入者アカウントに登録され、そのアカウントにバインドされた 1 つ以上の認証者が配信されます。IAL3 は、より高度な攻撃を制限し、高度な証拠の改ざん、盗難、否認、およびより高度なソーシャル エンジニアリング戦術から保護するように設計されています。

    ID プルーフの概要

    このセクションは規範的です。

    このセクションでは、本人確認と登録プロセスの概要と、申請者が主張した本人確認の解決、検証、検証をサポートするための要件について説明します。また、ID 証明プロセスの追加の側面に関するガイドラインも提供します。これらの要件は、要求された ID が現実世界に存在し、申請者がその ID に関連付けられた個人であることを確認することを目的としています。

    これらのガイドラインでは、解決、検証、検証を実現するための複数の方法と、ID 証明プロセスをサポートする複数の種類の ID 証拠が提供されています。CSPと組織は、さまざまな手段、能力、およびテクノロジーを持つ申請者のアクセスを促進するために、ID証明サービスとプロセスを実装する際にオプションを提供するべきです(SHOULD)。これらのオプションには、複数のタイプと組み合わせの身元証明の受け入れ、複数のデータ検証ソースのサポート、身元を検証するための複数の方法の有効化、複数の身元証明タイプの提供、および申請者に対する例外処理の提供(例:信頼できる査読者、申請者の参照)が含まれるべきです(SHOULD)。

    CSPは、提供される各ID証明オプション(ID証明の種類、検証ソース、支援メカニズムなど)に関連するリスクを評価し、必要に応じて不正行為を軽減するための制御を実装するものとします。少なくとも、CSPは、各オプションが総合的に同等の保証を提供するように設計するものとします(SHOULD)。

    要件は通常、特に明記されていない限り、CSP の責任としてこれらのガイドラインで表現されます。CSP 要件は、単一のエンティティによって実行される場合もあれば、すべての CSP の責任が果たされるように複数のコンポーネント サービスを含める場合もあります。

    本人確認と登録

    身元証明の目的は、身元証明プロセスに関与する申請者が、記載されたレベルの信頼度で本人であることを確認することです。この文書は、CSPが指定された保証レベルで申請者を身元証明するための3段階のプロセスを提示します。最初のステップは ID 解決であり、申請者が CSP がサービスを提供する母集団の一意の ID であり、実在の人物であるかどうかを判断するために、適切な ID 証拠と属性情報を収集することで構成されます。2 番目のステップは ID 検証で、最初のステップで収集された証拠と属性情報の信憑性、正確性、妥当性を確認します。3 番目のステップは本人確認で、身元証拠を提示する申請者が証拠が発行されたのと同じ個人であることを確認します。ほとんどの場合、指定された IAM に対する申請者の ID 証明に成功すると、CSP は申請者 (現在は ID サービスのサブスクライバー) の一意のサブスクライバー アカウントを確立し、1 つ以上のオーセンティケーターをアカウント内の証明済み ID にバインドできるようにします。

    ID 証明は、福利厚生またはサービスへの適合性または資格を判断するための組織のビジネス プロセスの一部にすることができますが、そのような決定はこれらのガイドラインの範囲外です。

    図 1 は、3 段階の ID 証明プロセスの例を示しています。

    図1.本人確認プロセス

    次の手順は、リモート無人 ID 証明の一般的なワークフローの例を示しており、規範的な処理ワークフロー モデルを表すことを意図したものではありません。

    1. 解決

      • CSP は、適切な強度の 1 つまたは 2 つの ID 証拠を収集します。
      • CSPは、提示された身元証拠に含まれるものを補足するために、必要に応じて申請者から追加のコア属性を収集します。

    2. 検証

      • CSP は、提示された証拠が本物で、正確で、有効であることを確認します。
      • CSP は、セクション 2.4.2.4 で説明されているように、ステップ 1 で取得した属性を信頼できる検証ソースと照合して検証します。

      • CSP は、承認された本人確認プロセスを採用して、申請者が提示された身元証拠の真の所有者であることを確認します。

      3つのID証明ステップが正常に完了すると、証明の通知が検証済みのアドレスに送信され、申請者は第5節で説明されているように、CSPの加入者アカウントに登録できます。加入者アカウントには、加入者とアカウントについて通信するために使用できる検証済み住所(電話番号、郵送先住所など)が少なくとも1つ含まれています。さらに、1つ以上の認証者が加入者アカウントの証明されたIDにバインドされます。

    ID プルーフの役割

    校正プロセス内で、個人ごとに異なる役割を果たすことができます。これらのガイドラインの一貫した実装をサポートするために、次の ID 証明ロールが定義されています。

    • プルーフィングエージェント:オンサイトまたはリモートのIDプルーフセッションに参加し、限定的なリスクベースの決定(たとえば、ID証拠を目視検査し、変更されていないと判断する)を行うように精査およびトレーニングされたCSPのエージェント。プルーフィングエージェントの最小トレーニング要件については、セクション3.13を参照してください。
    • 信頼できる審判: CSP、サードパーティ サービス、または証明書利用者 (RP) のエージェントで、申請者が定義された IAL 証明プロセスの予想される要件を満たすことができない場合に、申請者の ID 証明ケースに関してリスクベースの決定を下すように精査およびトレーニングされています。信頼できる査読者の要件は、セクション 3.15.1 に含まれています。証明エージェントとは対照的に、信頼できる査読者は、申請者が必要な身元証明を所有していない場合や、証拠の属性がすべて主張された身元と一致しない場合 (最近の名前や住所の変更など) など、例外処理シナリオをサポートするための追加のトレーニングとリソースを受けます。
    • 申請者紹介: 申請者の身元、申請者に関連する特定の属性、または個人の状況に関連する条件 (緊急状況、ホームレスなど) を保証できる申請者の代表者。申請者の紹介は、身元証明プロセスにおいて申請者に代わって行動するわけではありませんが、身元の主張を裏付けることはできます。申請者の紹介の要件は、セクション 3.15.3 に含まれています。
    • プロセスアシスタント: 本人確認プロセスで申請者を支援するためにサポートサービス(翻訳、文字起こし、アクセシビリティサポートなど)を提供するが、意思決定やリスクベースの評価はサポートしない個人。プロセスアシスタントは意思決定に関与していないため、これらのガイドラインにはこの役割に対する要件はありません。

    CSPは、上記のロールのうち、IDサービスに適用できるものを特定し、第3条で規定されている要件と期待に合致するトレーニングとサポートリソースを提供するものとします。

    ID 証明タイプ

    このドキュメントでは、ID 証明の種類は、CSP が採用するテクノロジ、通信チャネル、および ID 証明ロールの組み合わせによって定義されます。ID 証明の種類は、ID 証明が行われる場所と、プロセスに CSP のエージェントが参加するかどうかという 2 つの要因に基づいて特徴付けられます。

    • リモート無人ID証明:解決、検証、検証のプロセスが完全に自動化され、証明エージェントや信頼できる審判とのやり取りが不要な状態で実施されるID証明。校正プロセスで使用される場所とデバイスは、CSP によって制御されません。
    • リモート有人型 ID 証明: 申請者が校正エージェントまたは信頼できる審判との安全なビデオ セッションを通じて解決、検証、検証の手順を完了する ID 証明。校正プロセスで使用される場所とデバイスは、CSP によって制御されません。
    • オンサイトの無人 ID 証明: 個人が制御されたワークステーションまたはキオスクと対話する場合に実施される ID 証明ですが、校正エージェントまたは信頼できる審判との対話は必要ありません。このプロセスは完全に自動化されていますが、物理的な場所と CSP によって制御されるデバイス上にあります。
    • オンサイト有人の本人証明: 申請者が証明エージェントまたは信頼できる審判の立会いのもと、本人証明プロセス全体 (つまり、解決、検証、検証) を完了する物理的な環境で実施される本人証明。プルーフ エージェントまたは信頼できるレフェリーは、ユーザーと同じ場所に配置することも、キオスクまたはデバイスを介してユーザーと対話することもできます。物理的な場所とデバイスは、CSP によって制御されます。

    各保証レベルの要件は、CSP が必要に応じて、さまざまな保証レベルの要件を満たすために、ID 証明タイプのさまざまな組み合わせを実装できるように構成されています。CSPは、IDサービスによってサービスを提供する母集団のニーズと、サービスを使用するRPのリスク態勢に最もよく対応するID証明タイプの組み合わせを提供するべきです(SHOULD)。

    コア属性

    ID 証明プロセスには、解決、検証、検証を完了するために必要なものなど、ID 証明を達成するために必要な最小限の属性の提示と検証が含まれます。CSP は、セクション 3.1 で指定されているように、コア属性と見なす一連の属性を決定し、文書化します。CSP は、政府識別子を含めるものとします (SHALL shall the STATE)1また、任意のIALでのID証明のためのコア属性のセットに以下を含めるべきである(SHOULD)。

    • 名:申請者の名。
    • ミドルネームまたはイニシャル:該当する場合、申請者のミドルネームまたはイニシャル。
    • 姓:必要に応じて、申請者の姓または姓。
    • 生年月日:申請者が生まれた日。
    • 物理的またはデジタルの住所: 申請者が校正プロセスに関連する通信を受信できる物理的 (つまり、郵送先住所) またはデジタル (電話番号や電子メールなど) の住所。

    CSP と RP の要求に応じて、これらに追加の属性を追加しても構いません。CSPとRPは、信託契約および慣行に関する声明にすべてのコア属性を文書化するものとします(SHALL SHOULD)。プライバシーリスク評価に続いて、CSPは、ID証明を完了するために必要ではないが、他のRPビジネスプロセスをサポートする可能性のある追加の属性を要求してもよい(MAY)。追加の属性を要求するためのプライバシー要件の詳細については、セクション3.3を参照してください。

    ID 解決

    ID 解決には、ID 証明と、サービス対象の母集団の一意の ID を区別するために必要な最小限の ID 証拠と属性情報を CSP が収集することが含まれます。ID 解決は、潜在的な不正行為の初期検出を含む、ID 証明プロセス全体の出発点です。

    ID の証拠と属性

    ID 証拠は、主張された ID の現実世界の存在を裏付ける情報またはドキュメントです。身元証拠は、物理的 (運転免許証など) またはデジタル (モバイル運転免許証やデジタル アサーションなど) の場合があります。

    身元証明の収集は、身元確認プロセスをサポートし、1)身元証明申請者によるCSPへの身元証拠の提示と、2)提示された証拠が該当する強度要件を満たしているというCSPによる判断の2つのステップで構成されます。

    証拠強度の要件

    このセクションでは、各強度での身元証拠の要件を定義します。身元証拠の強度は、以下によって決まります。

    • 発行プロセスに関連する厳格さのレベル
    • 正確性と信頼性のチェックを含む、特定のレベルの信頼性で検証される機能
    • セクション2.5.1に示されている1つ以上の本人確認方法をサポートする機能

    この文書の付録 A は、強度によってグループ化された考えられる証拠タイプの非網羅的なリストを提供します。

    公正な証拠要件

    公正と見なされるには、身元証拠は次の要件をすべて満たすものとします。

    1. 顧客識別プログラム (CIP) 規則に基づいて顧客本人確認義務を負う金融機関によって発行された証拠や、モバイル ネットワーク オペレーター (MNO) に携帯電話アカウントを開設する手順など、主張された身元が対象に関連付けられていることを保証するために設計された正式な手順に従って、証拠の発行元が主張された身元を確認したという合理的な期待があります。
    2. 証拠には、郵便住所への配達、直接の発行、または保護されたリモート プロビジョニング プロセスを通じて、関連する個人に証拠が配信される発行プロセスがあります。
    3. 証拠には、主張された ID の名前が含まれています。
    4. 証拠には、少なくとも1つの参照番号、顔画像またはその他の生体認証特性、または関連する人物を一意に識別するのに十分な属性が含まれています。
    5. 証拠には、再現を困難にする物理的(セキュリティ印刷、光学的に可変の特徴、ホログラムなど)またはデジタル(デジタル署名されたアサーション、アドレス指定可能なSIMまたはESIMなど)のセキュリティ機能が含まれています。
    6. 証拠のコア属性は、セクション 2.4.2.4 で説明されているように、権威あるまたは信頼できる情報源が保持する情報に対して検証できます。
    7. 証拠は、セクション 2.4.2.2 に規定されているように、承認された方法を通じて検証できます。
    8. 証拠は、セクション 2.5.1 で説明されているように、本人確認プロセスをサポートできます。

    強力な証拠要件

    強力と見なされるためには、身元証拠は次の要件をすべて満たすものとします。

    1. 証拠の発行元は、主張された身元が被験者に関連付けられていることを保証するように設計された書面による手順(IAL2以上の身元証明など)に従って、主張された身元を確認したという合理的な期待があります。さらに、これらの手続きは、州、連邦政府、一部の規制業界などの規制機関または公的責任機関による定期的な監視の対象となります。
    2. 証拠には、郵便住所への配達、直接の発行、または保護されたリモート プロビジョニング プロセスを通じて、関連する個人に証拠が配信される発行プロセスがあります。
    3. 証拠には、主張された ID の名前が含まれています。
    4. 証拠には、参照番号または、関連する人物を一意に識別するその他の属性が含まれています。
    5. 証拠には、それが関係する人の顔画像またはその他の生体認証特性が含まれています。
    6. 証拠には、再現を困難にする物理的(セキュリティ印刷、光学的に可変の特徴、ホログラムなど)またはデジタルセキュリティ(デジタル署名されたアサーション、アドレス指定可能なSIMまたはESIMなど)の特徴が含まれています。
    7. 証拠のコア属性は、セクション 2.4.2.4 で説明されているように、権威ある情報源または信頼できる情報源によって保持されている情報に対して検証できます。
    8. 証拠は、セクション 2.4.2.2 に規定されているように、承認された方法を通じて検証できます。
    9. 証拠は、セクション 2.5.1 で説明されているように、本人確認プロセスをサポートできます。

    優れた証拠要件

    優れたものと見なされるためには、身元証拠は次の要件をすべて満たすものとします。

    1. 証拠の発行元は、主張された身元が被験者に関連付けられているという高い確信を持てるように設計された書面による手順 (IAL2 以上の身元証明など) に従って、主張された身元を確認しました。さらに、これらの手続きは、州、連邦政府、一部の規制業界などの規制機関または公的責任機関による定期的な監視の対象となります。
    2. ID 証拠には、暗号的に保護されている属性とデータ オブジェクトが含まれており、発行元によって適用されたデジタル署名の検証を通じて、承認された暗号化を使用して検証できます。
    3. 発行元は、被験者に、物理的な存在を確認する出席登録および本人確認プロセスに参加させました。
    4. 証拠には、郵便住所への配達、直接の発行、または保護されたリモート プロビジョニング プロセスを通じて、関連する個人に証拠が配信される発行プロセスがあります。
    5. 証拠には、主張された ID の名前が含まれています。
    6. 証拠には、関連する人物を一意に識別する参照番号が少なくとも 1 つ含まれています。
    7. 証拠には、それが関係する人の顔画像またはその他の生体認証特性が含まれています。
    8. 証拠には、再現を困難にする物理的(セキュリティ印刷、光学的に可変の特徴、ホログラムなど)またはデジタルセキュリティ(デジタル署名されたアサーション、アドレス指定可能なSIMまたはESIMなど)の特徴が含まれています。
    9. 証拠は、セクション 2.4.2.2 に規定されているように、承認された方法を通じて検証できます。
    10. 証拠は、セクション 2.5.1 で説明されているように、本人確認プロセスをサポートできます。

    ID の証拠と属性の検証

    身元証拠の検証の目的は、収集された身元の証拠が本物で有効であることを判断することです。属性検証の目的は、すべてのコア属性の正確性を確認することです。

    この文書では、「有効」という用語を使用して、証拠が有効期限が切れていたり、定められた期間外に発行されたりした場合でも、身元を証明するための有用な手段であり続けることができることを認識しています。CSP は、セクション 3.1 で説明されている CSP プラクティス ステートメントの一部として、期限切れの証拠に対処するためのポリシーを定義し、RP は、これがオンライン サービスへのアクセスに受け入れられるかどうかを判断します。

    身元証拠の検証には、提示された証拠を調べて、それが本物であること (つまり、偽造または改ざんされていない)、正確であること (つまり、証拠に関する情報が正しい)、および有効であること (つまり、有効期限が切れていないか、CSP が定義した発行または有効期限内) であることを確認することが含まれます。属性の検証には、提示された証拠から得られたものか、自己主張されたものかにかかわらず、コア属性の正確性を確認することが含まれます。次のサブセクションでは、証拠と属性の検証に許容される方法を示します。

    証拠の検証

    CSP は、次のことを確認することにより、提示された証拠の信憑性、正確性、および妥当性を検証するものとします (SHALL SHOULD)。

    • 証拠は正しい形式であり、ID 証拠の種類に関する完全な情報が含まれています
    • 証拠は偽造または改ざんされた形跡を示していません
    • 証拠には、物理的またはデジタルのセキュリティ機能が含まれています
    • 証拠の信憑性を判断するために必要なコア属性とデータフィールドは正確です

    証拠検証方法

    提示された証拠を検証するために許容される方法は次のとおりです。

    • 現場での身元証明のための訓練を受けた担当者による目視および触覚検査
    • 遠隔身元証明のための訓練を受けた担当者による目視検査
    • 適切なテクノロジーを使用した自動文書検証プロセス
    • デジタル証拠または属性データオブジェクトのソースと整合性の暗号化検証

    一部のデジタル証拠 (MNO/電話アカウントなど) では、視覚的または物理的に検証できる物理的な証拠がありません。信頼性は、発行者からのデジタル署名されたアサーションを検証したり、そのアカウント情報にアクセスできる属性検証サービスにクエリを実行したりするなど、発行元または信頼できるソースを使用して、そのアカウントと電話番号に関連付けられた ID 属性を検証することによって確認されます。

    属性の検証

    CSPは、身元証拠から得られたものか、申請者が自己主張したものかにかかわらず、すべてのコア属性(第2.2条)を、権威あるまたは信頼できる情報源で検証するものとします(第2.4.2.4条)。

    検証ソース

    CSP は、次の基準を満たす信頼できる情報源を使用するものとします (SHALL SHOULD)。

    信頼できる情報源は、ID 証拠または属性の発行ソースであるか、発行ソースによって管理されている情報に直接アクセスできます。発行元の例としては、運転免許証データの州自動車局や社会保障カードと番号の社会保障局などがあります。発行元への直接アクセスを提供または可能にする権威ある情報源の例としては、米国自動車管理者協会の運転免許証データ検証 (DLDV) サービスがあります。

    信頼できるソースは、信頼できるソースにトレースできる属性情報にアクセスできるか、複数のソースから取得した ID 属性情報を、正確性、一貫性、および最新性について相関関係があるものを保持します。信頼できる情報源は規制当局の監視の対象となります(公正信用報告法など)。

    本人確認

    本人確認の目的は、主張された検証済みの身元と、指定されたレベルの信頼度で身元証明プロセスに従事する実際の申請者との間のリンクを確立することです。言い換えれば、検証は、証拠を提示する申請者がその証拠の正当な所有者であることを保証するものです。

    本人確認方法

    CSPは、以下の方法の1つ以上を通じて、請求された身元と本人確認プロセスに従事する申請者との関連性を確認するものとします。

    • 確認コードの検証: 個人は、セクション 3.8 で指定された要件と一致して、確認コードの返却を通じて、身元証拠の管理を実証できます。
    • 認証およびフェデレーションプロトコル: 個人は、認証またはフェデレーションプロトコルを使用して、デジタルアカウント(オンライン銀行口座など)または署名されたデジタルアサーションの制御を実証できます。これは、直接行うことも、デバイスやリーダーに資格情報を提示することも、リモートの ID 証明セッション中に行うこともできます。
    • トランザクションの検証: 個人は、CSP と証拠の発行元との間で行われたマイクロトランザクション (金融口座の所有権を確認するための少額預金など) に基づいて値を返すことで、証拠の管理を実証できます。
    • 視覚的な顔画像比較 — オンサイト出席: 本人確認プロセス中に、校正エージェントと申請者が直接対話します。校正エージェントは、身元証明に提示された顔画像と、身元証明イベントに従事する申請者の顔との視覚的比較を実行します。
    • 視覚的な顔画像比較 — リモート有人またはリモート無人: 校正エージェントは、身元証明イベントに従事する申請者の顔画像に対して、身元証拠に提示された顔画像または発行元によって保存された顔画像の視覚的比較を実行します。校正エージェントは、身元証明プロセスの一部または全部で申請者と直接対話するか、キャプチャしたビデオまたは写真とアップロードされた証拠のコピーを使用して後で比較を行う場合があります。
    • 自動生体認証比較:自動生体認証比較(顔認識やその他の完全に自動化されたアルゴリズム駆動の生体認証比較など)は、オンサイトまたはリモートの身元証明イベントに対して実行できます。身元証拠上の顔画像またはその他の生体認証特性 (指紋、掌紋、虹彩および網膜パターン、声紋、静脈パターンなど) または信頼できる記録に保存されているものは、身元証明イベント中に申請者から収集された同等の生体認証サンプルと比較されます。

    ナレッジベース検証(KBV)またはナレッジベース認証は、本人確認に使用してはなりません。

    1. 政府識別子は、政府の記録 (社会保障番号、運転免許証番号、パスポート番号など) で申請者に関連付けられている一意の識別子です。↩

    Identity Service のドキュメントとレコード

    CSPは、定義されたIALを達成するために実装されるすべての本人確認プロセスを詳述する文書化された手順または慣行ステートメントに従って業務を遂行するものとします(SHALL SHOULD)。これらの文書化された手順には、少なくとも以下が含まれます。

    1. 提供された各保証レベルで CSP が ID 証明申請者に従う特定の手順を含む、完全なサービス説明

    3. 申請者がすべての要件を満たした後、本人確認プロセスがタイムリーに終了することを保証するための CSP のポリシー
    4. CSP が受け入れる証拠の種類と、証拠が CSP によって受け入れられるレベルの強度要件をどのように満たすかの正当化
    5. 2.1.2 条に規定されているように、身元証明の役割を担う担当者のトレーニングと資格要件を含む、身元証明の検証と検証に関する CSP のポリシーとプロセス
    6. CSP が証拠の検証と検証に採用する特定のテクノロジ
    7. 必要な IAL (セクション 3.15) に十分な身元証明が不足している申請者をサポートし、身元証明の例外とエラーに対処するための CSP のポリシーとプロセス
    8. CSP がコア属性と見なす属性 (セクション 2.2) と、それらの属性の検証に使用する権威ある信頼できるソース (セクション 2.4.2.4)
    9. データソース、統合ベンダー、生体認証アルゴリズムの変更など、サービスの変更を管理およびRPに伝達するためのCSPのポリシー
    10. 不正管理に対するCSPのアプローチ(第3.2条を参照)、不正の疑いまたは確認されたアカウントを特定して是正し、そのような情報をRPおよび影響を受ける個人に伝達するためのポリシーとプロセスを含む
    11. ユーザーの再確認を必要とする条件(アカウントの回復、アカウントの放棄、規制上の「再認定」要件など)に関するCSPのポリシー
    12. 定期的なレビューのタイミングや、最新のプライバシーリスク評価をトリガーする特定の条件など、プライバシーリスク評価を実施するためのCSPのポリシー(セクション3.3.1を参照))
    13. 採用されたテスト方法、定期的なレビューのタイミング、およびサイクル外のレビューを引き起こす特定の条件を含む、顧客エクスペリエンスを評価するためのCSPのポリシー(セクション3.4を参照))
    14. CSP が運用を停止したり、別の CSP に運用をマージまたは転送したりした場合の、そのようなすべてのデータの処理を含む、すべての個人データ、機密データ、および生体認証データの保持、保護、削除に関する CSP のポリシー
    15. SP 800-63 のセクション 3.5.2 に記載されている、パフォーマンス メトリックのレポートと更新に関する CSP のポリシー
    16. 加入者が死亡または無能力になった場合に加入者のアカウントにアクセスまたは削除するための CSP のポリシー (第 5 条を参照))

    CSPは、文書化された手順またはプラクティスステートメントを、IDサービスを使用するRPが利用できるようにするものとします(SHOULD)。CSPは、文書化された手順またはプラクティスステートメントの要約版を公開すべきです(SHOULD)。

    SP 800-63C では、信頼契約を使用して、フェデレーション関係における ID プロバイダー(IdP)、CSP、および RP 間の要件を定義する方法について説明します。CSP プラクティス ステートメントは、これらの契約に直接含めても構いません (MAY)。

    不正管理

    ID 証明プロセスの重要な側面は、ID 管理システムによって保護されている特典、サービス、データ、または資産にアクセスしようとする不正な試みを軽減することです。解決、検証、検証のプロセスは、多くの種類の攻撃を軽減するように設計されています。ただし、脅威環境が絶えず変化する中で、追加のチェックと制御を階層化することで、証明された ID の信頼性を高め、高度で新しいタイプの攻撃に対する追加の保護を提供できます。潜在的な不正行為のインスタンスを特定、検出、解決する機能は、CSP と RP にとって重要な機能です。

    CSPの不正管理

    1. CSPは、不正行為の特定、検出、調査、報告、および解決機能を提供する不正管理プログラムを確立および維持するものとします。このプログラムの特定の機能と詳細は、CSP プラクティス ステートメントに文書化されるものとします (SHOULD)。
    2. CSPは、実装前に、すべての不正チェックおよび不正軽減テクノロジーのプライバシーリスク評価を実施するものとします。
    3. CSPは、詐欺の被害者である、または身元証明プロセスへの関与を侵害しようとする試みであると信じる被験者に対して、自己報告メカニズムと調査能力を確立するものとします。
    4. CSPは、すべてのリモートプルーフ通信チャネルを分析して、リスクの高い指標(ブロックリストに登録されたプロキシやIPアドレスなど)を探すものとします(SHOULD)。
    5. CSPは、不合格の応募者が、権威あるまたは信頼できる情報源によって確認された情報と自己主張の情報の正確性を推測することを防ぐための措置を講じるものとします。1
    6. CSPは、不正チェックおよび不正軽減テクノロジーのパフォーマンスを監視して、不正リスクの軽減における継続的な有効性を確保するものとします(SHOULD)。
    7. CSP は、疑わしい、確認された不正事象を RP に伝達するための技術的またはプロセスベースのメカニズムを確立するものとします (SHOULD)。
    8. CSPは、申請者が死亡していないことを信頼できる、権威のある、または発行元に確認することにより、すべての本人確認プロセスについて死亡記録チェックを実施するものとします(SHOULD)。このようなチェックは、合成 ID 詐欺、盗まれた ID 情報の使用、側近者や親族による悪用の防止に役立ちます。
    9. CSPは、利用可能なID証明タイプ、選択されたテクノロジー、証拠、およびユーザーベースに基づいて、ID証明プロセスに対して次の不正チェックを実装すべきです(SHOULD)。
      • SIMスワップ検出:本人確認プロセスで使用した電話番号が、最近新しいユーザーまたはデバイスに移植されていないことを確認します。このようなチェックにより、電話またはデバイスが標的型攻撃によって侵害されたことを示すことができます。
      • デバイスまたはアカウントのテニュアチェック:電話サービスのサブスクリプションまたはその他のアカウント (電子メール アカウントなど) が大幅な変更や変更を行わずに存在した期間を評価します。このようなチェックにより、身元証明プロセスで使用されるデバイスまたは証拠の信頼性に対するさらなる信頼性が得られます。
      • 郵送先住所の確認:郵送先住所が既知の仮想郵便局 (PO) ボックスであるか、その他のリスクの高い特性を持っているかを判断します。
      • デバイスのフィンガープリンティング:デバイス フィンガープリンティング チェックを組み込んで、スケーリングおよび自動化された攻撃や登録の重複から保護します。デバイスフィンガープリンティングは、デバイスの一意の識別子(つまり、フィンガープリント)を作成するために、デバイスのハードウェアとソフトウェアの特性を収集および分析するプロセスです。
      • トランザクション分析:予想されるトランザクション特性 (IP アドレス、地理位置情報、トランザクション速度など) を評価して、より高いリスクや潜在的な詐欺イベントを示す可能性のある異常な動作やアクティビティを特定します。不正速度チェックは、特定の期間にわたるトランザクションの頻度とパターンを監視して、トランザクション データに関連する異常なアクティビティを特定します。このようなチェックは、スケーリングされた自動攻撃から保護できるだけでなく、特定の攻撃パターンがアイデンティティシステムで実行されているかどうかを示すことができます。
      • 不正指標チェック:記録 (報告された、確認された、または過去の不正イベントなど) を評価して、特定の申請者、申請者のデータ、またはデバイスに関連するリスクが高いかどうかを判断します。このようなチェックは、個人情報の盗難や侵害を示している可能性があります。そのような情報が商用プラットフォーム間で収集、集約、または交換され、RPおよびその他のCSPが使用できるようになる場合、ユーザーはプライバシーリスク評価に基づいてプライバシーへの影響を認識されるものとします(SHOULD)。これは、ユーザーのアクティビティを連邦政府の RP に報告するすべての Web サイトにも当てはまります。
    10. CSPは、不正対策の有効性を検証するために、独立したテスト(レッドチーム演習など)を定期的に採用すべきです(SHOULD)。
    11. CSPは、不正防止機能や意思決定にそのようなデータを織り込む際に、不正関連データの最新性を考慮するべきです(SHOULD)。
    12. 有人校正プロセスの場合、CSPは、不正の兆候を検出するために校正エージェントをトレーニングし、校正エージェントと信頼できる審査員に、さらなる処理と調査のために不正行為の疑いのあるイベントにフラグを立てるためのツールを提供するものとします。
    13. CSPの担当者が校正プロセスや決定に直接関与している場合は、共謀が可能です。CSPは、証明プロセスまたは決定に直接関与している、または介入できるCSP担当者が関与する共謀を検出および防止するために、インサイダー脅威制御を実装するものとします(SHOULD)。
    14. CSPは、[SP800-63C]のセクション4.8に記載されているように、共有シグナリングなどの方法を通じて、不正イベントをリアルタイムでRPに伝達すべきである(SHOULD)。
    15. CSPは、詐欺管理プログラムの一環としてKBVを採用しても構いません。
    16. CSPは、補償管理として不正対策を実装しても構いません(MAY)。これが行われると、これらはこれらのガイドラインの規範的ガイダンスからの逸脱として文書化され、統合前にデジタル ID 受諾声明 (DIAS) を通じてすべての RP に伝達されるものとします。DIASの詳細については、[SP800-63]の3.4.4節を参照してください。

    ID サービスの一部として人工知能(AI)または機械学習(ML)を採用する CSP は、該当する場合、[SP800-63] の第 3.8 条に規定される要件を遵守するものとします。

    RP不正管理

    1. RPは、CSPが不正データをやり取りし、伝達できる連絡先を確立するものとします。
    2. RPは、潜在的なプライバシーリスクまたは意図しない損害を特定するために、CSPの不正チェックおよび軽減技術のプライバシーリスク評価(第3.3.1条を参照)を実施するものとします。連邦機関の RP は、第 3.7 条に含まれる要件に従ってこれを実装するものとします。
    3. RP は、CSP との信頼契約に不正チェックおよび不正軽減テクノロジーに関する要件を含めるべきです (SHOULD)。
    4. RPは、CSPの不正管理プログラム、不正チェック、および不正技術の定期的なレビューを実施して、しきい値を調整し、不正事象の調査を確認し、不正管理の有効性と有効性を評価するものとします。
    5. RPは、CSPが補償または補足的な統制として展開したすべての不正対策を、内部のリスク許容度と受け入れに合わせてレビューするものとします。RP は、統合前に CSP の補償制御を独自の DIAS に記録するものとします (SHOULD)。
    6. 適用される法律および規制に従い、RPは、CSPおよびその他のパートナーが独自の不正の特定、軽減、および報告機能を改善できるように、肯定的な結果と否定的な結果の両方を含む不正の報告および調査の結果をCSPおよびその他のパートナーに伝達するメカニズムを確立すべきです。
    7. RPは、その使命、規制環境、システム、アプリケーション、データ、およびリソースと整合性のあるデジタルID管理のための不正管理プログラムを確立すべきです(SHOULD)。
    8. プライバシーリスク評価で特定されたリスクから情報を受けて、RPは、不正行為に対抗したり、他のビジネスプロセスをサポートしたりするために、CSPがコア属性として提供するもの以外の追加の属性を要求することもできます(MAY)。

    不正チェックの失敗の処理

    不正チェックと軽減テクノロジーの有効性は、使用されるデータソース、使用されるテクノロジー、そしておそらく最も重要な申請者集団など、多くの要因によって異なります。したがって、不正管理対策から生じる障害に対処する、適切に構造化され、文書化されたプロセスを持つことが重要です。これらの障害の処理には、次の要件が適用されます。

    1. CSPは、各不正チェックに関連するアクションとプラクティスを確立して文書化し、これらのアクションとプラクティスをRPに提供するものとします。
    2. CSPは、申請者が不正チェックおよび軽減技術に関連する問題を解決できるように、救済の手順を確立するものとします(SHOULD)。救済の詳細については、[SP800-63]のセクション3.6を参照してください。
    3. CSPは、無人のリモートプロセスで不正チェックに失敗した申請者に信頼できる審判サービスを提供すべきです(SHOULD)。無人のリモートプロセスで不正チェックに失敗した申請者に信頼できる審判が提供される場合、信頼できる審判には、リスクベースの意思決定プロセスに情報を提供するために、不正の失敗の結果の概要が提供されるものとします(SHOULD)。

    一般的なプライバシー要件

    次のプライバシー要件は、任意の IAL で ID サービスを提供するすべての CSP に適用されます。

    プライバシーリスク評価

    1. CSPは、本人確認と登録に使用されるプロセスについて、プライバシーリスク評価を実施し、文書化するものとします。2少なくとも、プライバシーリスク評価は、以下に関連するリスクを評価するものとします(SHOULD)。
      1. 身元証明、登録、または不正管理を目的とした個人情報(身元属性、生体認証、画像、ビデオ、スキャン、または身元証拠のコピーを含む)を処理すること
      2. CSPが申請者の身元を確認するために、ここに指定されている必須要件を超えて実行する追加の手順
      3. 法律または法的手続きを遵守する場合を除き、本人確認および登録の範囲外の目的での個人情報の処理
      4. 身元記録および個人情報の保存スケジュール
      5. アルゴリズム(AIやMLツールなど)によって集約または処理されたときに個人を特定するために使用できる非個人情報の処理
      6. CSP に代わって第三者サービスによって処理される個人情報
    2. CSPは、プライバシーリスク評価の結果に基づいて、収集または処理する個人情報の非関連付け可能性、予測可能性、管理可能性、機密性、完全性、および可用性を維持するために講じる措置を文書化するものとします。3このような措置を決定するにあたり、CSP は、NIST プライバシー フレームワーク [NIST-Privacy] や NIST SP 800-53 [SP800-53] などの関連するガイダンスと標準を適用すべきです (SHOULD)。
    3. CSP は、個人情報の処理に影響を与える ID サービスに変更を加えるたびに、プライバシー リスクを再評価し、プライバシー リスク評価を更新するものとします。
    4. CSPは、個人情報の収集と処理に関連する現在のリスクを正確に反映するために、その実践声明に記載されているように、プライバシーリスク評価を定期的に見直すものとします。
    5. CSPは、そのサービスを使用するすべてのRPがプライバシーリスク評価の要約を利用できるようにするものとします(MUST)。要約は、そのような RP が CSP サービスに関連するプライバシー リスクについて合理的な判断を下し、独自のプライバシー リスク評価を完了できるようにするのに十分な詳細であるものとします (MUST)。
    6. CSPは、加入者アカウントで保持されている個人情報の処理についてプライバシーリスク評価を実行するものとします(第5条を参照)。

    追加のプライバシー保護対策

    1. 個人情報の処理は、主張された身元の存在を検証し、主張された身元を申請者に関連付け、詐欺を軽減し、承認の決定を行うために使用できる属性をRPに提供するために必要な最小限に限定されるものとします。
    2. CSP は、CSP の ID サービスに関連する機密情報にアクセスできるすべての担当者およびサードパーティのサービス プロバイダーにプライバシー トレーニングを提供するものとします。
    3. CSP は、ID 解決に必要な場合、社会保障番号 (SSN) を属性として収集しても構いません (MAY)。SSN の知識は身元の証拠として機能するのに十分ではなく、証拠として使用する場合、社会保障カードの所持を確認するための許容される方法とは見なされません。SSN が連邦、州、または地方自治体の機関に代わって収集される場合、CSP は、適用法に従って収集の申請者に通知を提供するものとします。
    4. 収集時に、CSPは、属性および個人情報の収集目的について申請者に明示的に通知するものとします。このような通知には、個人情報と属性が本人確認プロセスを完了するために自発的であるか必須であるか、CSPが申請者の後続の加入者アカウントに保存する予定の特定の属性およびその他の機密データ、属性を提供しなかった場合の結果、および記録保持要件が設けられている場合の詳細が含まれます。 データの削除を要求したり、その他の形態の救済に従事したりする申請者の権利を含みます。
    5. CSPは、第7条のプライバシーに関する考慮事項に基づいて、申請者のプライバシーを保護する手法を実装すべきである(SHOULD)。

    一般的なカスタマーエクスペリエンス要件

    CSPは、IDサービスの要素を評価して、サービスを提供する人々のカスタマーエクスペリエンスの課題をもたらす可能性のあるプロセスとテクノロジーを特定します。顧客体験に対するリスクが特定された場合、CSPは、保証レベルとリスク態勢に合致したこれらの問題を軽減または排除する緩和策を積極的に採用します。

    次の要件は、任意の IAM で ID サービスを提供するすべての CSP に適用されます。

    1. CSPは、ID証明プロセスの要素を評価して、特にそれらの課題がCSPがRPが提供するすべてのユーザーにID証明サービスを一貫して提供することを妨げる場合、顧客エクスペリエンスの課題を引き起こす可能性のあるプロセスまたはテクノロジーを特定するものとします(SHALL SHOULD)。
    2. CSPは、ユーザーが直面する一般的な課題または問題に関する情報を含む、カスタマーエクスペリエンス評価の概要をRPに提供するものとします(SHOULD)。
    3. CSPは、評価の結果に基づいて、起こり得るアクセスの課題を軽減するために講じる措置を文書化するものとします。
    4. CSPは、カスタマーエクスペリエンスのリスクを定期的に再評価するものとし、CSPがカスタマーエクスペリエンスに影響を与えるプロセスまたはテクノロジーに影響を与えるIDサービスに変更を加えるたびに再評価するものとします。
    5. CSPは、これらのリスク評価への申請者の参加を義務付けないものとします。

    一般的なセキュリティ要件

    1. 第三者が関与する取引を含む、本人確認プロセス内の各オンライン取引は、認証された保護されたチャネルを介して行われるものとします(SHOULD)。
    2. CSPは、ボットの検出、軽減、管理ソリューションなど、ID証明プロセスに対して自動化された攻撃保護を実装するものとします。行動分析4;Web アプリケーション ファイアウォールの設定。ネットワークトラフィック分析。
    3. 本人確認プロセスの一環として収集されるすべての個人情報は、保存データの暗号化や認証された保護されたチャネルを使用した情報の交換など、情報の機密性と完全性を維持するために保護されるものとします。
    4. CSPは、NISTリスク管理フレームワーク[NIST-RMF]または同等のリスク管理ガイドラインに従って、IDサービスの運用に関連する情報セキュリティおよびプライバシーリスクを評価するものとします(SHOULD)。少なくとも、CSP は、IAL に関係なく、NIST SP 800-53 [SP800-53] の中程度のベースラインと一致する適切なコントロールを適用するものとします (SHOULD)。
    5. CSPは、[SP800-161]システムおよび組織向けのサイバーセキュリティサプライチェーンリスク管理慣行に規定されているように、サードパーティサービスの使用に関連するリスクを評価し、適切な管理を適用するものとします。

    救済要件

    1. CSPは、証明の失敗、遅延、困難、侵害された加入者アカウントの回復(詐欺や詐欺の結果など)など、本人確認プロセスから生じる申請者の苦情および問題を是正するためのメカニズムを提供するものとします(SHOULD)。
    2. これらの救済メカニズムは、申請者が簡単に見つけて使用できるものでなければなりません。
    3. CSPは、苦情または問題の解決を達成する際の有効性のメカニズムを評価するものとします(SHOULD)。

    救済の詳細については、[SP800-63]の第3.6節を参照してください。

    確認コードの要件

    このセクションには、確認コードの使用をサポートする CSP の要件が含まれています。

    確認コードは、申請者が将来の通信の目的で住所、電子メール アドレス、または電話番号にアクセスできることを確認するために使用されます。郵便番号または電話番号の住所に配信された確認コードは、第4.1.6項および第4.2.6項で説明されているように、IAL1および2で本人確認オプションとしても使用できます。

    これらの目的に使用される確認コードには、承認されたランダムビットジェネレータからの少なくとも10進数6桁(または同等の数字)が含まれているものとします([SP800-63B]のセクション3.2.12を参照)。確認コードは、手動入力用の数値または印刷可能なASCII表現、確認コードの表現を含む安全な(例えば、https)リンク、または確認コードを含む機械可読光学ラベル(例えば、QRコード)として提示されてもよい。

    確認コードは、最大で次の期間有効であるものとします。

    • 米国本土内の有効な住所に送付された場合は 21 日
    • 米国本土以外の有効な住所に送付された場合は 30 日間
    • 検証済みの電話番号(SMSまたは音声)に送信した場合は10分
    • 検証済みの電子メール アドレスに送信された場合は 24 時間

    使用時に、CSPは確認コードを無効にするものとします(SHALL SHALL SHOULD)。

    継続コードの要件

    継続コードは、不完全な本人確認または登録プロセスへの申請者のリンクを再確立するために使用されます。継続コードは、あるセッションを別のセッションに接続できる一時的なシークレットを提供します。一般的なシナリオでは、申請者がオンラインで本人確認プロセスを開始するが (リモート無人など)、対面 (オンサイト出席など) イベントを通じて完了する必要があります。このオンサイト サービスは、多くの場合、サードパーティによって、または確立された CSP 発行の認証子による申請者の認証をサポートするテクノロジーを持たないチャネルを通じて提供されます。申請者が ID 証明プロセスのすべてのステップで確立された認証システムを活用できる場合は、継続コードは必要ありません。

    そのため、申請者が 1 回のセッションで ID 証明を正常に行い、CSP の ID サービスに登録するために必要なすべての手順を完了できない場合、特に異なる ID 証明タイプを切り替える場合、CSP は継続コードを使用しても構いません (MAY)。継続コードは、オフラインで維持され (印刷または書き留められるなど)、以前の未完了のセッションへのリンクを再確立するために申請者が安全な場所に保存することを目的としています。CSPとの後続のセッションへの申請者の認証を容易にするために、CSPはまず、最初のセッションの停止前に申請者のために確立されたレコードまたはアカウントに認証者をバインドすべきである(SHOULD)。

    継続コードを使用する場合は、次の要件が適用されます。

    1. 継続コードはセッション内で配信されるべきですが、物理的な郵送先住所、電話番号、または電子メール アドレスに帯域外で配信されても構いません。
    2. 継続コードは、承認されたランダムビットジェネレータからの少なくとも64ビットを含むものとする(SHALL 3.2.12 of [SP800-63B]のセクションを参照)。
    3. 継続コードは、手動入力用の数値または印刷可能なASCII表現として、または機械可読の光学ラベル(例えば、QRコード)として提示されても構いません(MAY)。
    4. 継続コードの検証は、[SP800-63B] のセクション 3.2.2 に規定されているように、スロットリング要件に従うものとします (SHOULD)。
    5. 継続コードは、連邦情報処理標準 (FIPS) が承認した、または NIST が推奨する一方向関数を使用して、ハッシュ形式で保存されるものとします (SHOULD)。
    6. その使用に際して、CSP は継続コードを無効にするものとします (SHALL SHOULD)。

    申請者が継続コードを受け取ってから校正プロセスを完了できるようになるまでにかなりの時間が経過する可能性があるため、これらのガイドラインでは有効期限は定義されていません。これは、CSP がプロセス、テクノロジー、パートナーシップに基づいて定義する必要があります。

    3. 受取人が本人証明イベントへの参加を拒否した場合に取るべき措置について、連絡先情報を含む明確な指示を提供するものとします
    4. 組織またはCSPが収集した情報のセキュリティと機密性をどのように保護しているかに関する情報を提供するものとします(SHALL
    5. 受信者がIDサービスの加入者として負う責任に関する情報を提供するものとします(SHALL

    生体認証の使用に関する要件

    生体認証とは、顔の特徴、指紋、音声パターン、キーストロークパターン、スマートフォンの持ち方、画面の圧力、タイピング速度、マウスの動き、歩行などの生物学的および行動的特徴に基づいて個人を自動的に認識することを指します。これらのガイドラインで使用される生体認証データとは、他のアプリケーションまたはサービスパートナーへの生体認証データの送信を含む、取得、保存、または処理のあらゆる段階における生物学的および行動的特徴のアナログまたはデジタル表現を指します。これには、申請者からのライブ生体認証サンプル (顔画像、指紋など) と、証拠から得られた生体認証参照 (運転免許証の顔画像、身分証明書の指紋の詳細テンプレートなど) が含まれます。ID 証明プロセスに適用されるように、CSP は生体認証を使用して、個人が ID 証拠の正当な対象であることを確認したり、個人を新しい ID 証拠または資格情報にバインドしたり、重複排除の目的で行うことができます。これらの要件は、身元証明プロセスでの生体認証の使用に関連する追加のプライバシーへの影響にも対処します。

    次の要件は、ID 証明プロセスの一環として生体認証を採用する CSP に適用されます。

    1. CSPは、生体認証のデータが収集されるか、その保存と保護方法、適用される法律および規制に準拠した生体認証データを削除する方法など、生体認証のすべての使用について、明確で公開されている情報を提供するものとします。
    2. CSPは、すべての申請者から生体認証を収集および使用するために、明示的なインフォームドコンセントを得るものとします。
    3. CSPは、生体認証の使用に対する加入者の同意の記録を保存し、それを加入者のアカウントに関連付けるものとします(SHOULD)。
    4. CSPは、文書化され、公開されている削除プロセスと、すべての生体認証情報のデフォルトの保持期間を有するものとします(MUST)。保持期間は、CSP がサービスを提供する地域およびセクターに適用される規制、ポリシー、および法令と一致するものとします。
    5. CSP は、規制、法律、またはポリシーによって別段の制限がある場合を除き、加入者の要求に応じて、加入者の生体認証情報の削除を支持すべきです (SHOULD)。生体認証削除リクエストをサポートしない CSP は、ポリシーの規制、法定、またはリスクベースの正当な理由を公に文書化するものとします (MUST)。
    6. CSPは、生体認証および攻撃検出アルゴリズムを、独立したエンティティ(認定された研究所や研究機関など)によって、人口統計グループ全体のパフォーマンスを含むパフォーマンス特性について定期的にテストさせるものとします。さらに、CSP は、プロバイダーの更新スケジュールに基づいて生体認証アルゴリズムの内部テストを実施すべきです (SHOULD)。
    7. CSPは、システムの運用環境およびユーザーベースと実質的に類似した条件で、採用された生体認証技術のパフォーマンスおよび人口統計学的影響を評価するものとします(MUST)。ユーザーベースは、予想されるユーザーと、使用することが予想されるデバイスの両方によって定義されます。そのような評価に現実世界のユーザーが含まれる場合、ユーザーによる参加は任意であるものとします。
    8. CSP は、要求された ID に対する検証に 1 対 1 (1:1) の比較アルゴリズムが使用される場合、次のパフォーマンスしきい値を満たすものとします (SHOULD)。
      • 偽一致率: 1:10,000 以上
      • 偽の不一致率: 1:100 以上
    9. CSPは、プライバシーリスク評価に従って、解決または重複排除をサポートするために1対多(1:N)の識別を使用しても構いません(MAY)。1:N シナリオでは、CSP は、誤検知識別の最小パフォーマンスしきい値である 1:1,000 以上を満たすものとします (SHOULD)。
    10. 申請者が収集した生体認証特性をデータベースに対して 1:N で検索して、申請者がデータベースにすでに存在しているかどうか、場合によっては別の名前で存在しているかどうかを判断します。偽陽性識別率 (FPIR) とは、生体認証システムが別の人物を一致として誤って識別する 1:N 検索の割合を指し、これは偽陽性の結果です。1:1,000 のパフォーマンス メトリックは、1,000 回の検索ごとに 1 回以下で誤検知の結果が発生することを意味します。この要件を実証するテストでは、現在または意図された運用サイズ(N)の90%以上のギャラリーを採用するものとする(MUST)。
    11. 解決、重複排除、または不正検出の目的で 1:N 生体認証を利用する CSP は、自動検索結果を確認し、結果が誤検知 ID ではないことを確認するための手動レビューなしに、ユーザーの登録を拒否してはなりません (例: 同じ CSP で異なるアカウントの顔写真を送信する双子)。
    12. 生体認証技術は、さまざまな人口統計タイプの申請者に対して、全人口のパフォーマンスよりも25%以下劣るパフォーマンスを提供するものとします。たとえば、母集団全体で測定された誤不一致率 (FNMR) が 0.006 の場合、特定の人口統計グループの FNMR は 0.0075 を超えることはできません。同様に、母集団全体の偽一致率 (FMR) が 0.0001 の場合、各人口統計グループの FMR は 0.000125 を超えることはできません。生体認証システムは、固定しきい値で構成されるものとします。人口統計グループごとにしきい値を変更することは現実的ではありません。考慮される人口統計学的カテゴリーには、性別、年齢、およびこれらの要因が生体認証パフォーマンスに影響を与える場合の肌の色が含まれるものとします。
    13. すべての生体認証性能テストは、人口統計テストを含め、ISO/IEC 19795-1:2021 および ISO/IEC 19795-10:2024 に準拠するものとします。
    14. CSPは、生体認証アルゴリズムのパフォーマンスおよび生体認証システムの運用テスト結果の結果を公開するものとします(MUST)。CSP は、結果がこれらのガイドラインで定義されたメトリックおよびテストされた人口統計グループ全体でのパフォーマンスを示している場合、これらのテスト結果を要約形式で提供しても構いません (MAY)。

    申請者から生体認証特性を収集するCSPには、次の要件が適用されます。

    1. CSPは、生体認証特性が他の被験者ではなく申請者から収集されたことを合理的に保証する方法で生体認証特性を収集するものとします。

    視覚的な顔画像比較の要件

    本人確認をサポートする校正エージェントと信頼できる査読者は、提示された証拠の顔の肖像画を、その証拠に表された身元を主張する申請者と比較できる必要があります。そのため、CSPがこの視覚的な顔画像比較を検証オプションとして提供する場合、次の要件が適用されます。

    1. 校正エージェントと信頼できる審判員は、視覚的な顔画像比較を実施するための訓練を受けるものとします。このトレーニングには、顔の特徴、固有の特徴、および申請者と提示された証拠との間の一致または不一致のその他の指標を特定するための技術と方法が含まれるものとします。
    2. 校正エージェントと信頼できる査読者は、視覚的な顔画像比較を行う能力について評価されるものとします。さらに、校正エージェントと信頼できる査読者は、必要に応じて毎年再評価され、修復されたトレーニングを受けるものとします。トレーニングは、応募者を親戚、双子、および似たような外見を持つ個人の画像と比較するなど、潜在的な現実世界の攻撃シナリオを反映するように設計されるものとします。
    3. CSPは、高品質の画像フィード、高解像度モニター、画像分析ソフトウェアなど、正確な比較をサポートするリソースを使用して、リモートアテンドトランザクション中に視覚的な顔比較を行う校正エージェントおよび信頼できる査読者を提供するものとします。
    4. CSPは、視覚的画像比較のためのトレーニングおよび評価手順を文書化し、要求に応じてRPが利用できるようにするものとします。
    5. これらの要件は、自動生体認証比較の失敗(例えば、解像度または重複排除のために実施された1:Nチェックの失敗)の手動レビューとして行われる視覚的な顔画像比較に適用されるものとします(MUST)。

    物的証拠の検証要件

    物的証拠の検証は、光学キャプチャと検査(文書認証または「文書認証」と呼ばれることが多い)によって、または訓練を受けた校正エージェントまたは信頼できる審判による目視検査によって実施できます。CSP は、どちらかまたは両方のプロセスを使用して、身元証拠の信頼性を評価できます。

    次の要件は、文書の信頼性を判断する目的で光学キャプチャと検査を採用する CSP に適用されます。

    1. 自動証拠検証テクノロジーは、次のパフォーマンス指標を満たすものとします(SHOULD)。
      • 文書の虚偽受理率(DFAR)が0.1以下6
      • 文書の誤拒否率(DFRR)が0.1以下7
    2. 証拠に機械可読ゾーン(MRZ)またはバーコードが存在する場合、光学キャプチャおよび検査は、一貫性を保つためにMRZデータを証拠の印刷データと比較するものとします(SHOULD)。
    3. CSPは、検証プロセス中にドキュメントのライブキャプチャを実装するものとし、パッシブまたはアクティブのドキュメントプレゼンスチェック(ドキュメントのライブネスとも呼ばれます)を実装するものとします(SHALL SHOULD)。ライブキャプチャ技術は、ドキュメントが物理的に存在すること、およびID証明セッション中にキャプチャされた画像が操作されたデジタルコピーではないことを確認します。改変メディア(デジタルで生成されたビデオまたは証拠の画像)の挿入を防ぐための追加要件については、セクション3.14を参照してください。
    4. CSPは、運用環境およびシステムのユーザーベースによって提示された証拠の種類と実質的に類似した条件で、採用された光学キャプチャおよび検査技術のパフォーマンスを評価するものとします。これらのテストは、CSPが光学キャプチャおよび検査技術を使用して検証することを許可する、利用可能なすべてのID証拠タイプを説明するものとします(SHOULD)。加入者の文書、個人情報、または画像がテストの一部として使用される場合、それは自発的に行われ、加入者の通知と同意を得るものとします。
    5. CSPは、独立した機関(認定された研究所や研究機関など)によって、そのパフォーマンス特性について、証拠検証技術を定期的にテストすべきです(SHOULD)。
    6. CSPは、テストの結果を一般に公開するものとします(SHOULD)。

    これらの要件は、物理的な ID 証拠の画像をキャプチャして検証するテクノロジに適用されます。これらは、証拠自体に組み込まれているPKIまたはその他の暗号化技術に依存する検証手法には適用されません。

    次の要件は、文書の信頼性を判断する目的で、訓練を受けた校正エージェントまたは信頼できる審判による証拠の目視検査を採用するCSPに適用されます。

    1. 校正エージェントと信頼できる査読者は、CSPによって裏付けられるあらゆる形態の証拠を視覚的に検査するためのトレーニングを受け、リソースを提供するものとします。このトレーニングには以下が含まれます。
      • 証拠タイプの本物のレイアウトと地形
      • 物理的なセキュリティ機能(隆起した文字、ホログラフィック機能、マイクロプリントなど)
      • 特徴を評価するための手法(例えば、使用するツール、触覚検査が必要な場所、特定の特徴を表示するために必要な操作)
      • 改ざんの一般的な兆候(ラミネートの損傷、画像の改ざんなど)
    2. 校正エージェントと信頼できる審判員は、トレーニングに基づいて証拠を目視検査する能力について評価されるものとします。さらに、証明エージェントと信頼できる査読者は、毎年、または証拠検証プロセスに対する重大な新しい脅威が特定され、必要に応じて是正トレーニングされるたびに再評価されるものとします。
    3. 校正エージェントおよび信頼できる審判員には、身元証拠の種類に応じて、証拠の目視検査をサポートするための特殊なツールおよび機器(拡大鏡、紫外線、バーコードリーダーなど)が提供されるものとします。
    4. 遠隔手段を介して目視検査を行う校正エージェントおよび信頼できる査読者には、提示された証拠を効果的に検査できるように十分に高品質の画像をサポートするデバイスとインターネット接続が提供されるものとします。このような場合、視覚的検証は、証拠の信憑性に対するさらなる信頼性を提供する自動文書検証技術によってサポートされるべきである(例えば、有人リモートセッションに先立って証拠を提出して検証する)。
    5. CSPは、証拠の目視検査のためのトレーニングと評価手順を文書化し、要求に応じてRPが利用できるようにするものとします。

    身元証拠の数と順列が異なる可能性があるため、これらのガイドラインはセキュリティ機能の包括的なリストを提供しようとはしていません。CSP は、受け入れる ID 証拠の種類に固有の証拠検証トレーニングを提供する必要があります。

    デジタルインジェクション防止と偽造メディア検出

    有人型と無人型の両方のリモートID証明プロセスに対する新たな攻撃の多くは、デジタルインジェクション攻撃と、ますます効果的で利用可能な生成AIツールを組み合わせています。これらの AI ツールは、申請者の画像やビデオ、証拠 (ディープフェイクなど) を含むメディアを作成または変更し、校正エージェントによって行われる自動文書検証プロセス、生体認証操作、視覚的比較を無効にするために使用されます。インジェクション攻撃は、キャプチャポイント(デバイスなど)と、比較やその他の操作を行う要素(アルゴリズムを実行するサーバー、プルーフィングエージェントが使用するワークステーションなど)の間に、改変または偽造されたメディアを挿入します。

    攻撃がリモートの光学キャプチャおよび検査コンポーネント、自動化された生体認証メカニズム、またはリモート有人プロセスで使用されるビデオシステムに対する攻撃のいずれであっても、あらゆる種類のリモートIDプルーフは、何らかの形でこれらの形式の攻撃に対して脆弱です。

    キャプチャされたサンプルを使用して生体認証比較を実行しても、これらの攻撃を防ぐことはできません。ただし、ライブドキュメントキャプチャとプレゼンテーション攻撃の検出メカニズムは、実行可能な偽造メディアの挿入をより困難にすることで、インジェクションおよび偽造メディア攻撃からある程度の保護を提供します。メディアは、申請者エンドポイントと CSP 比較コンポーネント間の通信チャネルに挿入する必要があるだけでなく、偽造されたメディアは、CSP によって実装された受動的または能動的なプレゼンテーション攻撃検出メカニズムを十分に打ち負かす必要もあります。ただし、これらのメカニズムでさえ、この種の攻撃の考えられるすべてのケースに対処するには十分ではありません。

    次の要件は、証拠検証、リモート生体認証キャプチャ、およびビデオセッションに光学キャプチャおよび認識ツールを使用するすべてのリモートID証明プロセス(つまり、無人および有人)に適用されます。

    1. CSPは、プルーフプロセス中にデジタルメディアが本物のセンサーによって作成されているという信頼性を高めるための技術的制御を実装するものとします(例:仮想カメラ、デバイスエミュレーター、またはジェイルブレイクされたデバイスの存在を検出します)。
    2. CSPは、本人確認プロセス中に提出されたすべてのデジタルメディアを分析して、潜在的な変更、操作、改ざん、または偽造のアーティファクトおよび指標がないか分析するものとします(SHOULD)。自動画像分析アルゴリズムは、利用可能な攻撃アーティファクト(すなわち、偽造および操作された画像およびビデオ)および本物のメディアに対してテストされ、パフォーマンスのベースラインを提供し、システムによって生成される誤検知および誤検知の予想される率を決定するものとします。テストされた利用可能な攻撃アーティファクトの種類とそれに対応する偽陰性率は文書化され、要求に応じてRPが利用できるようにするものとします。メディアのアルゴリズム分析と自動意思決定は、検出エラーに対処するために手動レビューによって強化されるべきです(SHOULD)。
    3. CSPは、リモートID証明プロセス中のデータの交換にのみ、認証された保護されたチャネルを使用するものとします(SHALL SHOULD)。
    4. CSP は、すべてのキャプチャ シナリオについて、偽造または操作されたメディアを検出する受動的な手段を導入すべきです (SHOULD)。
    5. CSP は、リモート ID 証明プロセスの一環としてデジタル メディアの送信に使用されるデバイスの信頼性を高めるために、キャプチャ センサーを認証するか、デバイス構成証明を実装する必要があります (SHOULD)。
    6. CSPは、偽造デジタルメディアの作成に使用されることが知られている生成AIアルゴリズムとディープフェイクツールのシグネチャについて、デジタルメディアを分析すべきです(SHOULD)。

    次の追加要件は、リモート有人コレクションのシナリオに適用されます。

    1. CSPは、校正エージェントと信頼できるレフリーをトレーニングして、操作されたメディアの兆候(高遅延、同期の問題、一貫性のない肌の色と解像度など)を探すものとします。
    2. CSPは、偽造または操作されたメディアが検出される可能性を高めるために、ランダムな「ヒューマン・イン・ザ・ループ」の手がかりをキャプチャ・プロセスに導入するものとする(例えば、ユーザーの動きを要求したり、ユーザーがキャプチャ・センサーと顔の間で物体を移動させるよう要求したりすることによって)。

    例外とエラー処理

    本人確認プロセス全体を通じて、エラーや障害が発生する可能性のあるポイントが数多くあります。標準的な本人確認ワークフローのこのような例外には、プロセスの失敗(ユーザーが必要な証拠を所有していない場合など)、技術的な失敗(統合サービスが利用できない場合など)、およびユーザーエラーによる失敗(申請者がリモート検証ツールを使用して身元証拠の鮮明な画像をキャプチャできない場合など)が含まれます。

    ID 証明サービスのアクセシビリティを高め、顧客エクスペリエンスの課題に対処するために、CSP は、エラーの処理と例外の処理に関する運用プロセスを文書化するものとします。これらの文書化されたプロセスには、IAL 1 および 2 の要件を満たすことができない申請者をサポートするための信頼できる審査員の提供が含まれるべきです (SHOULD)。さらに、CSPは、申請者の属性、条件、または身元を保証できる申請者参照の使用を支持すべきです(SHOULD)。

    信頼できる審判員の要件

    信頼できる査読者は、特定の IAL の通常の身元証明プロセスを使用して身元を証明できない個人の身元証明と登録を容易にすることにより、オンライン サービスへのアクセスを増やすために使用されます。信頼できる審判の支援を必要とする可能性のある個人の例の非網羅的なリストには、必要な身元証拠を所持していない人、または取得できない人、障害のある人、高齢者、ホームレスを経験している人、オンラインサービスまたはコンピューティングデバイスへのアクセスが制限されている人、銀行口座がない人、または信用履歴が限られている人、 個人情報の盗難の被害者、自然災害によって避難または影響を受けた個人、および18歳未満の子供。信頼された審査員は、CSP、サード パーティ、または RP によって提供できます。信頼できる審査員が使用される場合は、次の要件が適用されます。

    1. CSPは、信頼できる審判サービスの利用可能性と、そのようなサービスがどのように取得されるかを一般の人々に通知するものとします(SHOULD)。
    2. CSPは、第3.1条に規定されているように、その実践方針の一部として、信頼できる審判員の使用に関する書面による方針と手順を確立するものとする(SHOULD)。
    3. CSPは、信頼できる査読者を訓練し、認定して、応募者が固有の状況に基づいて身元を証明できるように、リスクベースの決定を下すものとします。少なくとも、そのようなトレーニングには以下が含まれるものとします。
      1. 一般的なテンプレート、セキュリティ機能、レイアウト、地形などの文書の識別と検証 (セクション 3.13 を参照))
      2. 損傷、改ざん、改ざん、捏造、偽造などの不正な文書の兆候 (セクション 3.13 を参照))
      3. 提出された書類と申請者を確認するための顔画像比較 (セクション 3.12 を参照))
      4. 申請者が示したソーシャルエンジニアリングの指標(苦痛、混乱、強制など)
      5. 証拠を目視検査し、視覚的な顔画像を比較する信頼できる審判の能力の年次レビュー (セクション 3.12 を参照))
    4. CSPは、信頼できる審判が使用された理由(例:自動化されたプロセスの失敗、申請者の要求、確立された例外ポリシー)、信頼できる審判の身元、提示された証拠、完了したプロセス(例:検証または検証)、および信頼できる審判の決定を含む、信頼できる審判が関与する本人確認セッションの記録を確立するものとします。 否定的であれば、その理論的根拠。
    5. CSP は、オンサイト有人またはリモート有人セッションのいずれかに対して、信頼できる審判サービスを提供してもよい (MAY)。これらのセッションは、プルーフ イベントの IAL に基づくこれらのプルーフ タイプの要件と一致するものとします (MUST)。

    信頼できる審判員の使用

    信頼できる審判は、他の手段で身元証明を完了できない人に重要な道を提供します。ただし、校正プロセス内で発生する可能性のある障害の数を考慮すると、CSPは、信頼できるレフリーを自社のサービス提供内で適用できる用途を定義することが不可欠です。信頼できる査読者を ID 証明プロセスに統合する定義には、次の要件が適用されます。

    1. CSP は、信頼できるレフリーの使用に適格な例外と障害の種類を文書化するものとします (SHOULD)。
    2. CSPは、自動検証プロセス(生体認証比較など)の失敗に対して、信頼できる審判サービスを提供すべきである(SHOULD)。
    3. CSPは、コア属性の不一致やレコードソースへの申請者の不在など、自動検証プロセスの完了に失敗した場合、信頼できるレフリーサービスを提供すべきです(SHOULD)。CSP がこの目的のために信頼できる審査員を提供する場合、次の要件が適用されます。
      1. CSPは、コア属性またはコア属性の変更を裏付けるために使用できる追加の証拠タイプに関するポリシーを提供するものとします(SHOULD)。
      2. 信頼できる査読者は、証拠が許容する最大限の範囲で、追加の証拠タイプの信憑性を審査するものとします(SHOULD)。
      3. 検証を裏付ける信頼できる記録がない場合、信頼できる査読者は、コア属性の一貫性を裏付けるために利用可能な最も強力な証拠と、追加の証拠の属性を比較しても構いません。
      4. コア属性と権威ある記録に部分的な不一致がある場合、信頼できる審判は、主張された属性値の正当性を裏付ける証拠(例えば、最近の引っ越しや名前の変更)を審査するものとする(SHOULD)。

    申請者の推薦状の要件

    申請者紹介とは、申請者の身元証明、属性、または本人確認を完了する申請者の能力に関連する状況を保証するために、申請者の身元証明に参加する個人です。申請者の参照は、CSPの代理人ではなく、他の形式の証拠、検証、および検証が利用できない場合に、本人確認の完了を支援するのに十分な知識を持つ申請者の代表者です。

    IAL1またはIAL2で申請者の紹介がサポートされている場合、次の要件が適用されます。

    1. CSPは、申請者の推薦状の許容性、および推薦状と申請者の関係に関する要件を一般の人々に通知するものとします。
    2. CSPは、第3.1条に規定されているように、その実践声明の一部として申請者の参照を使用するための書面によるポリシーと手順を確立するものとします。
    3. CSPは、申請者を対象とした同じ以上のIALへの申請者の参照を身元証明するものとします。CSPは、セクション3.3.1で要求されているように、申請者の参照を身元証明するために収集、記録、および保持された情報をプライバシーリスク評価に含めるものとします(SHOULD)。
    4. CSPは、加入者アカウントにおける申請者参照の使用を記録し、申請者参照と申請者との関係の記録を維持するものとします(MUST)。
    5. RPは、プルーフィングイベントのための申請者の参照を除外または含めることに伴う適用性、ビジネス要件、および潜在的なリスクを決定するために、リスク評価を実施するものとします。

    申請者の紹介の用途

    申請者の参照は、CSPおよびRPサービスの状況とコンテキストに応じて、本人確認プロセスで申請者をサポートするためにいくつかの異なるアクションを取ることができます。CSPは、オンラインサービスへのリスクが許す場合、申請者の参照の使用を提供するべきです(SHOULD)。CSPが申請者の参照の使用を許可する場合、CSPおよびそのサービスを使用するRPは、契約または信託契約において、申請者の参照の許容されるすべての使用を文書化するものとします(SHOULD)。

    1. 出願人の参照は、証拠および属性の検証プロセスの一環として、出願人に関連する1つ以上の主張されたコア属性を保証しても構いません。
    2. 申請者の参照は、十分な身元証拠がない場合、申請者の身元を保証することができます(MAY)。
    3. 申請者の参照は、身元証明プロセスに関連する申請者の特定の状態またはステータス(ホームレス、災害シナリオなど)を保証しても構いません。

    この情報は、身元証明イベントに関連するリスク判断をサポートすることを目的としています。ステータスまたは給付金の適格性を確立するための申請者の参照ステートメントの使用は、これらのガイドラインの範囲外です。

    すべての場合において、CSPは、申請者がプロセスで果たした役割の記録を確立し、適用される法的および規制上の要件をサポートするのに十分なこれらの行動を文書化するものとします。これには以下が含まれる場合があります。

    1. 申請者の参照によって行われた声明と主張をキャプチャして記録する
    2. 出願人参照のデジタル署名または物理署名のキャプチャ
    3. 申請者の推薦状の法的および責任への影響に関する同意と承認の取得

    CSPは、申請者参照としての参加から生じる可能性のある法的および責任の影響に関する明確で理解しやすい情報を申請者参照に提供するものとします(SHOULD)。

    申請者参照関係の確立

    多くの場合、申請者と申請者の紹介者との関係を確認するには、ビジネス上、法律上、または詐欺防止上の理由があります。リスク評価によってそのような手順が必要であると判断された場合は、次の要件が適用されるものとします。

    1. CSPとRPは、申請者の参照関係確認プロセスの要件を確立し、契約または信託契約に文書化するものとします。
    2. CSPは、関係確認プロセスを開始する前に、申請者が利用できる関係の許容可能な証拠のリストを作成するものとします(MUST)。
    3. CSPは、申請者の関係の証拠(公証された委任状、専門家証明書など)を要求するものとします(MUST)。
    4. 申請者の身元証明に成功すると、CSP は、申請者参照と申請者の関係を確認するために使用される証拠をサブスクライバー アカウントに記録するものとします。

    未成年者との交流に関する要件

    Identity Assurance レベル 1 の要件

    IAL1のID証明プロセスでは、悪意のある攻撃者によるIDに対する不正な主張を検出するためのさまざまな許容可能な技術が可能になり、ユーザーの採用を促進し、正当なユーザーの拒否を最小限に抑え、アプリケーションの離脱を減らします。生体認証マッチングの使用 (例: 申請者の顔画像と提供された証拠に基づく顔の肖像画の自動比較) は、IAL1 ではオプションです。

    校正タイプ

    1. IAL1 ID プルーフは、セクション 2.1.3 で説明されているように、任意のプルーフ タイプを通じて提供できます。
    2. CSPは、プルーフタイプとそれに記載された要件を組み合わせて、ハイブリッドプロセスを作成することができます。たとえば、CSP は、検証が行われるリモート有人セッションの前に、リモート無人 ID 証明検証プロセスを活用する場合があります。このような手順が組み合わされた場合、CSPはハイブリッドプロセスを文書化し、採用されている各校正タイプに適用可能な要件がどのように満たされているかを記載するものとします(MUST)。

    証拠収集

    IAL1 での身元証明のために、CSP は以下を収集するものとします (SHALL SHALL )。

    1. 1 ピース:
      • デジタルで検証できる、または顔の肖像画またはその他の生体認証を含む公正な証拠、または
      • STRONG エビデンスまたは
      • 優れた証拠。

    属性コレクション

    CSP は、少なくとも 1 つの政府識別子を含むすべてのコア属性を収集するものとします (SHALL )。検証済みの証拠は、ID 属性の優先ソースです。提示された身元証拠が、CSPがコア属性と見なすすべての属性を提供しない場合、CSPは、申請者が自己主張する属性を収集しても構いません(MAY)。

    証拠の検証

    提示された各証拠は、次のいずれかの方法を使用して検証されるものとします。

    • デジタルセキュリティ機能(アサーションやデータの署名など)を問い合わせることにより、デジタル証拠の信憑性を確認する
    • 第3.14条に記載されているように、物理的なセキュリティ機能を検出できる自動スキャン技術を使用して、物理的証拠の信憑性を確認する
    • リアルタイムまたは非同期プロセスを使用した校正エージェントによる目視検査により、提示された証拠の物理的セキュリティ機能の整合性を確認する(オフラインの手動レビューなど)
    • 現場でのプルーフエージェントによる物理的および触覚的検査による物理的セキュリティ機能の完全性の確認

    属性の検証

    1. CSP は、すべてのコア属性と政府識別子を、信頼できるまたは信頼できる情報源に対して検証して、正確性を判断するものとします (SHOULD)。
    2. CSPは、さまざまな情報源(提示された証拠、自己主張、権威ある、または信頼できる情報源など)から取得した属性の一貫性を評価するべきです(SHOULD)。
    3. CSP は、提示された ID エビデンス上の参照番号を検証すべきです (SHOULD)。

    検証要件

    CSPは、次のいずれかの方法を使用して、1つの証拠の申請者の所有権を確認するものとします(MUST)。

    • 証拠に関連付けられた検証済みのアドレスに配信された確認コードを返送する申請者の能力の確認
    • 検証済みの金融口座または同様の口座に配信されたマイクロトランザクションの価値を返す申請者の能力を確認する
    • AAL2/FAL2 以上と同等の認証およびフェデレーション プロトコルを正常に完了して、身元証拠に関連するアカウントにアクセスする申請者の能力を確認する
    • 自動比較による申請者の顔画像と証拠上の顔の肖像画の比較
    • 出願人の顔画像を、オンサイト有人セッション(つまり、校正エージェントとの対面)、リモート出席セッション(つまり、校正エージェントとのライブビデオ)、または非同期プロセス(つまり、校正エージェントによる視覚的比較)中に、証拠または証拠に関連する記録の顔の肖像画と視覚的に比較します。比較が後で非同期に実行される場合、CSP は PAD とパッシブまたはアクティブ ドキュメントの存在チェックを実装して、送信またはキャプチャ イベント中にライブ アプリケーションと物理的なドキュメントの両方が存在するという信頼性を高めるものとします。
    • 自動化された手段を使用して、身元証拠または証拠に関連する記録に表され、または保存されている顔画像を、申請者から提供されたライブサンプルと比較する

      1. [SP800-63B] の Sec. 4.1.3 で定義されているオーセンティケーター タイプの要件と一致する加入者提供のオーセンティケーターのリモート登録

      3. オーセンティケーターの配布またはオンサイト登録
    2. オーセンティケーターがユーザーとの 1 つの保護されたセッションの外部にバインドされている場合、CSP は、次のいずれかの方法を使用して、意図したサブスクライバーの存在を確認するものとします (SHALL SHOULD)。
      1. 継続コードの返却
      2. 校正時に収集した生体認証との比較

    ID 保証レベル 2 の要件

    IAL2 の ID 証明には、IAL1 に関連するなりすまし攻撃やその他の ID 証明エラーをより適切に軽減するための追加の証拠、検証、検証要件が含まれています。IAL2 は、さまざまな種類の ID 証明 (リモート無人、リモート有人など) を通じて実現でき、IAL2 での本人確認は、生体認証の使用の有無にかかわらず実現できます。このセクションでは、IAL2の結果と要件に合わせるための3つの異なる経路、IAL2検証 - 非生体認証経路、IAL2検証 - 生体認証経路、IAL2検証 - デジタル証拠経路について説明します。

    校正タイプ

    1. IAL2 ID 証明は、第 2.1.3 条に記載されているように、任意の ID 証明タイプを通じて提供できます。
    2. CSPは、ID証明タイプとそれに記載された要件を組み合わせて、ハイブリッドプロセスを作成することができます(MAY)。たとえば、CSP は、検証が行われるリモート有人セッションの前に、リモート無人 ID 証明検証プロセスを活用する場合があります。このような手順が組み合わされた場合、CSPはハイブリッドプロセスを文書化し、採用されている各校正タイプに適用可能な要件がどのように満たされているかを記載するものとします(MUST)。

    IAL2 での ID 証明のために、CSP は以下を収集するものとします。

    1. FAIRの証拠1枚と強力な証拠1枚、または
    2. 2つの強力な証拠または
    3. 優れた証拠の1つ。

    属性コレクション

    CSP は、少なくとも 1 つの政府識別子を含むすべてのコア属性を収集するものとします (SHALL )。検証済みの証拠は、ID 属性の優先ソースです。提示された身元証拠が、CSPがコア属性と見なすすべての属性を提供しない場合、CSPは、申請者が自己主張する属性を収集しても構いません(MAY)。

    証拠の検証

    1. 提示された FAIR または STRONG の各証拠は、次のいずれかの手法を使用して検証されるものとします。
      1. デジタルセキュリティ機能(アサーションやデータの署名など)を問い合わせることにより、デジタル証拠の信憑性を確認する
      2. 物理的なセキュリティ機能を検出できる自動スキャン技術を使用して、物的証拠の信憑性を確認します
      3. リアルタイムまたは非同期プロセス(オフラインの手動レビューなど)を使用した校正エージェントによる目視検査を通じて、物理的なセキュリティ機能の完全性を確認する
      4. 現場のプルーフエージェントによる物理的および触覚的な検査による物理的なセキュリティ機能の完全性の確認
    2. 優れた証拠の各部分は、デジタル署名の検証およびトラストアンカーへのトラストチェーンの検証を含む、証拠の内容と発行元の暗号検証を通じて検証されるものとします。この方法を使用して検証できない優れた証拠は、上記の手法のいずれかを使用して検証できる場合、強力な証拠と見なされる可能性があります。

    属性の検証

    1. CSP は、次のいずれかによってすべてのコア属性を検証するものとします (SHALL SHALL SHOULD)
      1. 政府の識別子やその他のコア属性を信頼できる情報源と比較して、正確性を判断する
      2. 発行元の公開鍵を使用して、SUPERIOR 証拠に含まれるデジタル署名された属性の正確性を検証する
    2. CSPは、さまざまな情報源(提示された証拠、自己主張、権威ある、または信頼できる情報源など)から取得した属性の一貫性を評価するべきです(SHOULD)。
    3. CSP は、提示された ID エビデンス上の参照番号を検証すべきです (SHOULD)。

    検証要件

    検証経路は、関連するポリシーと整合性を持って実装され、保護されるオンラインサービスのユースケース、母集団、脅威環境に対応するべきです(SHOULD)。CSPは、IAL2検証に複数の経路を展開すべきであり、望ましい結果を達成するために経路を組み合わせてもよい(MAY)。

    複数の検証経路を提供する CSP は、IAL2 を達成するためにどの経路がたどったかを加入者レコードに記録し、アサーション、API、または信頼契約の一部としてその情報を RP が利用できるようにするものとします。非生体認証経路を使用する場合、CSP は、郵送された確認コードまたは申請者と証拠との視覚的比較が検証に使用されたかどうかをさらに記録するものとします。

    IAL2検証 — 非生体認証経路

    IAL2 非生体認証経路は、申請者から提供された生体認証サンプルの自動比較を使用しない検証方法を提供します。この経路には、生体認証データの収集と検証 (たとえば、校正エージェントによって実行される身元証拠に含まれる顔画像との視覚的比較) が含まれる場合がありますが、そのような比較は自動ではなく手動の手段によって行われます。自動生体認証比較の使用を必要としない追加の検証方法も、セクション 4.2.6.2 で指定されたデジタル証拠経路の要件に含まれています。IAL2 でオプションとして提供されている場合、CSP は、ID サービスを使用するすべての RP に非生体認証経路の使用を通知するものとします。

    1. リモート有人、リモート無人、およびオンサイト無人の本人証明の場合、CSP は、提示されたすべての身元証拠の申請者の所有権を確認するものとします (MUST)。オンサイトでの身元証明の場合、CSPは、提示された身元証拠の最も強力な部分の申請者の所有権を確認するものとします(MUST)。
    2. IAL2 で FAIR の証拠を検証するために承認された非生体認証方法には次のものがあります。
      1. 証拠に関連付けられた検証済みの住所 (住所、電話番号など) に配信された確認コードを申請者が返送できることを確認する
      2. 申請者の顔画像を、提示された証拠(学生証や従業員証など)または証拠に関連する記録の顔の肖像画と視覚的に比較するオンサイト出席セッション(つまり、校正エージェントとの対面)、リモート出席セッション(つまり、校正エージェントとのライブビデオ)、または非同期プロセス(つまり、校正エージェントが別の時間に行った視覚的比較)
    3. IAL2 で STRONG および優れた証拠を検証するために承認された非生体認証方法には、次のものがあります。
      1. 証拠から取得され、信頼できる情報源で検証された物理的な住所 (つまり、郵便住所) に配信された確認コードを返送する申請者の能力を確認する
      2. 出願人の顔画像を、オンサイト出席セッション(つまり、校正エージェントとの対面)、リモート出席セッション(つまり、校正エージェントとのライブビデオ)、または非同期プロセス(つまり、校正エージェントによる異なる時間に行われた視覚的比較)中に、提示された証拠または証拠に関連する記録の顔の肖像画と視覚的に比較します。比較が後で非同期に実行される場合、CSP は、キャプチャされた ID 証明イベント中にライブ申請者と物理的な文書の両方が存在するという信頼性を高めるために、PAD およびパッシブまたはアクティブのドキュメント プレゼンス チェックを実装するものとします (SHOULD)。

    確認コードを物理的なアドレスに配信することと、信頼できるソースでアドレスを検証するという要件を組み合わせることで、ID 証明プロセスに対する大規模で大規模な攻撃に対する合理的な抑止力が得られ、攻撃者の価値実現までの時間に大きな影響を与えます。このため、郵送された確認コードは、生体認証や視覚的比較が失敗した場合、または代替手段が必要な場合(たとえば、加入者のテクノロジーやサービスへのアクセスが制限されている場合)に、IAL2での身元証明のための実行可能なオプションです。このような配信方法は、側近や家族による傍受や、その他のスキーム(メール転送詐欺など)に対して依然として脆弱です。このような攻撃の可能性と影響が高いと評価する組織は、他の検証方法を提供するか、追加の緩和制御を適用する必要があります。

    IAL2検証 — デジタルエビデンス経路

    IAL2 デジタル エビデンス パスウェイにより、個人は、デジタル資格情報 (デジタル身分証明書と呼ばれることもあります) やデジタル アカウントなどのデジタル形式の証拠を検証プロセスの一部として使用できます。

    1. リモート有人、リモート無人、およびオンサイト無人の本人証明の場合、CSP は、提示されたすべての身元証拠の申請者の所有権を確認するものとします (MUST)。オンサイトでの身元証明の場合、CSPは、提示された身元証拠の最も強力な部分の申請者の所有権を確認するものとします(MUST)。
    2. IAL2 で承認された FAIR 証拠のデジタル証拠検証方法には、次のものが含まれます。
      1. 検証済みの口座に配信されたマイクロトランザクション値を返す申請者の能力を確認する (例: 権威あるまたは信頼できる情報源によって検証された申請者が所有する当座預金口座)
      2. デジタル証拠に関連付けられた検証済みのデジタルアドレス(MNO/電話アカウントなど)に配信された確認コードを返送する申請者の能力を確認する
      3. AAL2/FAL2 と同等の認証およびフェデレーション プロトコルを正常に完了して、身元証拠に関連するアカウントにアクセスする申請者の能力を確認する
    3. IAL2 で承認された STRONG 証拠のデジタル証拠検証方法には、身元証拠に関連するアカウントにアクセスするために、AAL2/FAL2 以上と同等の認証およびフェデレーション プロトコルを正常に完了する申請者の能力を確認することが含まれます。
    4. IAL2 で承認された SUPERIOR 証拠のデジタル証拠検証方法には、ローカル活性化因子の使用と暗号的に検証可能な属性バンドルの提示による申請者の証拠の所有の確認が含まれます。

    この検証方法は、加入者制御のウォレットやスマートカードのPKI資格情報など、ローカル認証イベントを可能にする優れた証拠に対して実行可能です。これらの機能をサポートしていない優れた証拠 (ePassport など) は引き続き使用できますが、他の経路のいずれかを通じて検証する必要があります。

    IAL2検証 — 生体認証経路

    IAL2生体認証経路は、申請者から提供された生体認証サンプルの自動比較をサポートします。

    1. リモート有人、リモート無人、およびオンサイト無人の本人証明の場合、CSP は、提示されたすべての身元証拠の申請者の所有権を確認するものとします (MUST)。オンサイトでの身元証明の場合、CSPは、提示された身元証拠の最も強力な部分の申請者の所有権を確認するものとします(MUST)。
    2. IAL2 生体認証経路で使用するための FAIR、STRONG、および優れた証拠を検証するための承認された方法には、次のものがあります。
      1. 自動化された手段を使用して、身元証拠または証拠に関連する記録に表され、または保存されている顔画像を、申請者から提供されたライブサンプルと比較する
      2. 身元証拠または証拠に関連する記録に保存されている顔画像以外の生体認証特性を、申請者から提供されたライブサンプルと比較するための自動化された手段を使用する

    セクション 2、3、および 4 で指定された要件を超えるリモート無人 ID 証明の追加要件はありません。

    リモートアテンド要件

    1. ビデオセッション中、申請者は、校正プロセスの各ステップの間、校正エージェントの視野に留まるものとします。
    2. 動画の品質は、証拠の検査やユーザーと証拠の比較など、検証および検証プロセスで必要な手順をサポートするのに十分であるものとします。
    3. プルーフエージェントは、記録されたセッション中に発生する操作、強制、またはソーシャルエンジニアリングの兆候を特定するようにトレーニングを受けるものとします。
    4. CSPは、第3.3.1条に定義されているプライバシーリスク評価に従って、詐欺の防止および訴追の目的でビデオセッションを録画および維持することができます(MAY)。CSP がビデオ セッションを録画する場合、次の追加要件が適用されます。
      1. CSPは、録画セッションを開始する前に、申請者に録画を通知するものとします。
      2. CSP は、録画セッションを開始する前に、申請者から同意を得るものとします (MUST)。

    5. CSPは、第3.14条に定義されているように、インジェクション保護および修正されたメディア制御を実装するものとします(SHALL SHOULD)。
    6. CSPは、潜在的な不正行為のイベントにフラグを立てる方法またはメカニズムをプルーフエージェントに提供するものとします(SHOULD)。

    オンサイト有人要件

    1. CSPは、オンサイトの本人確認セッションが実施される物理的な環境を提供するものとします(SHOULD)。
    2. すべてのデバイスは、マルウェア保護、管理者固有のアクセス制御、およびソフトウェア更新プロセスを含む、FISMAの中程度の制御に匹敵する適切なベースラインセキュリティ機能によって保護されるものとします(SHOULD)。
    3. CSPプルーフエージェントは、オンサイトセッション中に発生する操作、強制、またはソーシャルエンジニアリングの兆候を特定するように訓練を受けるものとします。
    4. CSPは、第3.3.1条に定義されているプライバシーリスク評価に従って、詐欺の防止および訴追の目的でビデオセッションを録画および維持することができます(MAY)。CSP がビデオ セッションを録画する場合、次の追加要件が適用されます。
      1. CSPは、録画セッションを開始する前に、申請者に録画を通知するものとします。
      2. CSP は、録画セッションを開始する前に、申請者から同意を得るものとします (MUST)。

    オンサイト無人要件

    1. すべてのデバイスは、CSP 担当者による監視および/または物理的およびデジタルによる改ざん防止機能によって、改ざんから保護されるものとします。
    2. すべてのデバイスは、マルウェア保護、管理者固有のアクセス制御、およびソフトウェア更新プロセスを含む、FISMAの中程度の制御に匹敵する適切なベースラインセキュリティ機能によって保護されるものとします(SHOULD)。
    3. すべてのデバイスは、改ざん、改造、または損傷を防ぐために、訓練を受けた技術者によって定期的に検査されるものとします。
    4. CSPは、第3.3.1条に定義されているプライバシーリスク評価に従って、詐欺の防止および訴追の目的で、オンサイトの無人ID証明セッションのビデオ録画を録画および維持することができます(MAY)。CSP がビデオ セッションを録画する場合、次の追加要件が適用されます。
      1. CSPは、録画セッションを開始する前に、申請者に録画を通知するものとします。
      2. CSP は、録画セッションを開始する前に、申請者から同意を得るものとします (MUST)。

    オーセンティケーターの初期バインディング

    CSP の ID システムで申請者 (現在の加入者) に対して一意の加入者アカウントが確立されると、1 つ以上の認証子を加入者のアカウントに関連付ける (つまり、バインドする) ことができます。アカウント回復の必要性を最小限に抑えるために、CSP は加入者に少なくとも 2 つの個別の認証手段をバインドすることを奨励すべきです (SHOULD)。加入者アカウントの詳細については Sec. 5 を、オーセンティケーターのバインドの詳細については [SP800-63B] の Sec. 4.1.2.1 を参照してください。

    1. CSPは、申請者が以下のいずれかの方法を使用して認証器をバインドする機能を提供するものとします(SHALL SHOULD)。
      1. [SP800-63B] の Sec. 4.1.3 で定義されているオーセンティケーター タイプの要件と一致する加入者提供のオーセンティケーターのリモート登録
      2. 検証済みアドレスへの物理認証システムの配布
      3. オーセンティケーターの配布またはオンサイト登録
    2. オーセンティケーターがユーザーとの 1 つの保護されたセッションの外部にバインドされている場合、CSP は、次のいずれかの方法を使用して、意図したサブスクライバーの存在を確認するものとします (SHALL SHOULD)。
      1. 継続コードの返却
      2. 校正時に収集した生体認証との比較

    アイデンティティ保証レベル3

    IAL3は、IAL2で要求される手順にさらなる厳格さを加え、なりすましやその他の形態のID詐欺からIDとRPをさらに保護するために、生体認証情報の比較、収集、保持の使用を含む追加の特定のプロセスの対象となります。さらに、IAL3での身元証明は、セクション2.1.2に記載されているように、オンサイトで実行され、校正エージェントが立ち会います。

    校正タイプ

    IAL3 の本人確認は、オンサイトで出席した場合にのみ提供されるものとします。プルーフエージェントは、申請者と同じ場所に配置するか、CSPが制御するキオスクまたはデバイスを介してIDプルーフセッションに参加してもよい(MAY)。

    証拠収集

    IAL3での本人確認のために、CSPは以下を収集するものとします(SHALL SHALL MUST)。

    1. FAIRの証拠1枚と強力な証拠1枚、または
    2. 2つの強力な証拠または
    3. 優れた証拠の1つ。

    属性要件

    1. CSP は、少なくとも 1 つの政府識別子を含むすべてのコア属性を収集するものとします (SHALL )。検証済みの証拠は、ID 属性の優先ソースです。提示された身元証拠が、CSPがコア属性と見なすすべての属性を提供しない場合、CSPは、申請者が自己主張する属性を収集してもよい(MAY)。
    2. CSPは、アカウントの回復と否認防止をサポートし、証明と発行のプロセスに同じ参加者が存在するという高いレベルの信頼性を確立するために、本人確認プロセス中に申請者から生体認証サンプルを収集して保持するものとします(MUST)。CSPは、使用するモダリティと、加入者アカウントがそのような更新を正当化するのに十分な期間存続する可能性に基づいて、ユーザーの生体認証を定期的に再登録することを選択しても構いません(MAY)。

    証拠の検証

    1. 提示された FAIR または STRONG の各証拠は、次のいずれかの手法を使用して検証されるものとします。
      1. デジタルセキュリティ機能(アサーションやデータの署名など)を問い合わせることにより、デジタル証拠の信憑性を確認する
      2. 物理的なセキュリティ機能を検出できる自動スキャン技術を使用して、物的証拠の信憑性を確認します
      3. リアルタイムまたは非同期プロセス(オフラインの手動レビューなど)を使用した校正エージェントによる目視検査を通じて、物理的なセキュリティ機能の完全性を確認する
      4. 現場のプルーフエージェントによる物理的および触覚的な検査による物理的なセキュリティ機能の完全性の確認
    2. 優れた証拠の各部分は、デジタル署名の検証およびトラストアンカーへのトラストチェーンの検証を含む、証拠の内容と発行元の暗号検証を通じて検証されるものとします。この方法を使用して検証できない優れた証拠は、上記の手法のいずれかを使用して検証できる場合、強力な証拠と見なされる可能性があります。

    属性の検証

    1. CSP は、次のいずれかによってすべてのコア属性を検証するものとします (SHALL SHALL SHOULD)
      1. 政府の識別子やその他のコア属性を権威ある情報源または信頼できる情報源と比較して、正確性または信頼性を判断する。
      2. 発行元の公開鍵を使用して、SUPERIOR 証拠に含まれるデジタル署名された属性の正確性を検証する
    2. CSPは、さまざまな情報源(提示された証拠、自己主張、権威ある、または信頼できる情報源など)から取得した属性の一貫性を評価するべきです(SHOULD)。
    3. CSP は、提示された ID エビデンス上の参照番号を検証すべきです (SHOULD)。

    検証要件

    1. CSPは、以下のいずれかの方法で、最も強力な証拠(強力または優越的)の申請者の所有権を確認するものとします(MUST)。
      1. 物理的なデバイスまたはアプリケーション(モバイル運転免許証など)に対して正常に認証できる申請者の能力を確認し、デジタルで保護および送信された顔のポートレートを申請者と比較する
      2. 自動比較による申請者の顔画像と提示された証拠の顔の肖像画の比較
      3. オンサイト参加セッションまたは遠隔参加セッション中に、申請者の顔画像を提示された証拠の顔の肖像画と視覚的に比較する
      4. 身元証拠または証拠に関連する信頼できる記録に保存された生体認証を、申請者から提供されたサンプルと自動比較すること

    オンサイト アテンド要件 - コロケーション エージェント

    1. CSPは、オンサイトのID証明セッションが実施される安全な物理的な環境を提供するものとします。
    2. CSPは、申請者から生体認証を収集するためのセンサーとキャプチャデバイスを提供するものとします。
    3. CSP は、プルーフィングエージェントに、収集された生体認証のソースを、非天然材料 (パテ、接着剤など) の存在について表示させるものとします。
    4. CSPは、検定業者に、サンプルが申請者から収集され、他の情報源から収集されていないことを確認する方法で生体認証サンプルを収集させるものとします。
    5. CSPは、校正エージェントおよび信頼できる審判員が活用するすべての情報システムおよび技術が、校正施設の物理的管理を含む、少なくともFISMAの中程度または同等のレベルの管理と一致して保護されることを保証するものとします。
    6. CSPプルーフエージェントは、オンサイトセッション中に発生する操作、強制、またはソーシャルエンジニアリングの兆候を特定するように訓練を受けるものとします。
    7. CSPは、第3.3.1条に定義されているプライバシーリスク評価に従って、詐欺の防止および訴追の目的でビデオセッションを録画および維持することができます(MAY)。CSP がセッションを記録する場合、次の追加要件が適用されます。
      1. CSPは、録画セッションを開始する前に、申請者に録画を通知するものとします。
      2. CSP は、録画セッションを開始する前に、申請者から同意を得るものとします (MUST)。
      3. CSP は、すべてのビデオ レコードの保持スケジュールと削除プロセスを公開するものとします (SHOULD)。
    8. CSP は、プルーフ エージェントに、イベントまたはアクションを潜在的な詐欺として個別にフラグを立てる方法またはメカニズムを提供するものとします。

    オンサイト有人要件 - キオスクベース

    CSPは、エージェントと申請者が同じ場所(つまり、同じ部屋)にいない、プルーフィングエージェントと対話するリモート手段を提供してもよい(MAY)。たとえば、申請者は、ビデオを介してセッションにリモートで参加するプルーフエージェントを使用して、CSP制御のキオスクと対話することができます。このような場合、第4.3.7条のオンサイト有人身元証明に規定されている要件に加えて、以下の要件が適用されます。このドキュメントの以前のバージョンでは、このアプローチは監視付きリモート ID プルーフ(SRIP)と呼ばれていました。

    1. CSPは、申請者との高解像度ビデオ送信を通じて、本人確認セッション全体を監視するものとします。
    2. CSP は、本人確認プロセスの証拠収集、証拠検証、および検証手順について、ライブ校正エージェントが申請者とリモートで参加するものとします。解決と登録のための属性のデータ入力は、ライブプルーフエージェントの存在なしに実行してもよい(MAY)。
    3. CSP は、証拠収集、証拠検証、および検証手順中に申請者が行ったすべてのアクションが、リモート校正エージェントに明確に表示されるように要求するものとします (SHOULD)。
    4. CSPは、すべてのデジタル検証および証拠の検証(チップまたはワイヤレス技術など)が、統合されたスキャナーおよびセンサー(組み込み指紋リーダーなど)によって実行されることを要求するものとします(SHOULD)。
    5. プルーフィングエージェントと申請者の間の相互作用をサポートするために使用されるすべてのデバイスは、CSP担当者または監視デバイス(カメラなど)による監視、および物理的およびデジタルの改ざん防止機能を通じて、改ざんから保護されるものとします。
    6. プルーフィングエージェントと申請者の間の相互作用をサポートするために使用されるすべてのデバイスは、マルウェア保護、管理者固有のアクセス制御、およびソフトウェア更新プロセスを含む、少なくともFISMAの中程度の制御に匹敵する適切なベースラインセキュリティ機能によって保護されるものとします(MUST)。
    7. 校正剤と申請者の間の相互作用をサポートするために使用されるすべてのデバイスは、改ざん、変更、または損傷を阻止するために、訓練を受けた技術者によって定期的に検査されるものとします。

    校正の通知

    IAL3 での本人確認が正常に完了すると、CSP は、第 3.10 条に規定されているように、申請者の検証済みの住所に証明の通知を送信するものとします。CSP は、申請者の住所に校正の通知を送付すべきです (SHOULD)。

    オーセンティケーターの初期バインディング

    1. CSPは、プルーフィングエージェントとのオンサイト有人インタラクション中に、加入者の初期認証器を配布または登録するものとします(SHALL SHOULD)。

    サブスクライバーアカウント

    サブスクライバーアカウント

    CSPは、登録時からアカウント閉鎖時まで、IDシステム内のアクティブなサブスクライバーごとに一意のサブスクライバーアカウントを確立し、維持するものとします。CSP は、各サブスクライバーを ID サービス内の一意の ID として記録し、そのアカウントに関連付けられているすべてのオーセンティケーターの記録を維持するために、サブスクライバー アカウントを確立します。

    CSP は、各加入者アカウントに一意の識別子を割り当てるものとします (SHALL SHOULD)。識別子は、CSP のシステムによってランダムに生成され、ユーザー集団内の一意性を確保し、該当する場合は RP とのフェデレーションをサポートするのに十分な長さとエントロピーを持つべきです (SHOULD)。識別子は、[SP800-63C]と一致して、アサーションの生成におけるサブジェクト識別子として使用してもよい(MAY)。

    少なくとも、CSP は、各サブスクライバー アカウントに次の情報を含めるものとします (SHOULD)。

    • サブスクライバーアカウントに関連付けられた一意の識別子
    • サブスクライバに対して確立されたサブジェクト識別子 (RP 固有のサブジェクト識別子を含む)
    • サブスクライバーに対して完了した ID 証明手順の記録には、以下が含まれます。
      • 身分証明書の種類と発行者
      • 校正の種類(つまり、リモート無人、リモート有人、オンサイト有人、オンサイト無人)
      • 使用される検証および検証方法
      • 信頼できるレフェリーまたはその他の例外処理プロセスの使用
      • 出願人参照の一意の識別子を含む、出願人参照の使用
    • 加入者の身元証明のために最大IALが正常に達成されました
    • 生体認証を含む、加入者アカウントのライフサイクル全体にわたる申請者に関する情報の収集と処理に関連する申請者の同意契約の記録
    • 現在サブスクライバーアカウントにバインドされているすべてのオーセンティケーター(登録時または登録後に登録されているかどうか)
    • RP アクセスをサポートするために、ID 証明プロセス中または後続のトランザクションで検証された属性

    CSPは、本人確認されていない申請者(仮名アカウントなど)の加入者アカウントを登録して確立し、そのようなステータスを加入者アカウントに記録することができます。

    サブスクライバーアカウントへのアクセス

    CSP は、加入者が加入者アカウント内の情報を認証し、アクセスする機能を提供するものとします (SHOULD)。

    個人情報を含む加入者アカウントの場合、この機能は、加入者アカウントに登録されている認証システムを使用した AAL2 または AAL3 認証プロセスを通じて実現されるものとします (SHOULD)。

    加入者アカウントのメンテナンスと更新

    CSP は、加入者が加入者アカウントで情報の更新を要求する機能を提供するものとします (SHOULD)。CSP は、サブスクライバーが非コア属性を直接更新するメカニズムを提供しても構いません (MAY)。

    物理アドレスを除き、CSP は、加入者アカウントで管理されているコア属性情報に対する変更を検証するものとします (SHALL SHOULD)。CSP は、加入者の物理アドレスが CSP によってコア属性と見なされる場合、加入者の物理アドレスへの変更を検証する必要があります (SHOULD)。

    CSPは、加入者アカウント内の情報に加えられた更新を加入者に通知するものとします(SHALL SHOULD)。

    CSPは、加入者が加入者アカウント内の情報への不正アクセスまたは侵害の可能性を報告する機能を提供するものとします(SHOULD)。

    加入者アカウントの停止または終了

    CSPは、次のいずれかが発生した場合、加入者アカウントを速やかに一時停止または終了するものとします。

    • サブスクライバーは、CSP のサブスクライバー アカウントを終了することを選択します。
    • CSP は、サブスクライバー アカウントが侵害されたと判断します。
    • CSP は、サブスクライバーが CSP ID サービスに参加するためのポリシーまたは規則に違反したと判断します。
    • CSP は、CSP によって確立されたポリシーまたはルールに従って、サブスクライバー アカウントが非アクティブであると判断します。
    • CSP は、権威あるソースからサブスクライバの死亡の通知を受け取ります。
    • CSP は、加入者のアカウントを終了するように指示する法的文書を受け取ります。
    • CSP は、ID システムとサービスの操作を停止します。

    CSP は、アカウントが停止または終了された場合、加入者に通知するものとします (SHOULD)。このような通知には、アカウントが停止または終了された理由、再アクティブ化または更新のオプション、およびアカウントが誤って停止または終了されたとサブスクライバーが判断した場合の救済オプションに関する情報が含まれるものとします(SHOULD)。

    CSPは、その慣行に関する声明(第3.1条を参照)に記載されているように、記録の保持および廃棄の要件に従って、アカウント終了後の加入者アカウント記録からすべての個人情報を削除するものとします(SHALL SHOULD)。

    データ侵害の通知

    CSPレコードのデータ侵害が発生した場合、CSPは、個人情報が不正アクセスにさらされた可能性のある加入者に通知を提供するものとします(MUST)。このような通知には、侵害に関する情報と、加入者がアカウントへのアクセスを回復または維持し、個人情報の不正開示から保護するために取るべき措置が含まれるものとします。CSPは、そのような通知を加入者の検証済みアドレスに可能な限り迅速に送信するものとします(SHALL SHOULD)。

    複数のサブスクライバ アカウントのシナリオ

    一部の CSP は、さまざまなロールまたはペルソナを果たしながら、1 人のユーザーが CSP と対話する機能をサポートする必要があります。たとえば、加入者は、CSP がサポートするさまざまなサービス間で、自分自身として、またビジネスの代表として対話できます。別のシナリオは、商用 RP と連邦 RP の両方にサービスを提供し、サービス間の分離を維持する必要がある CSP です。不正行為を制限し、冗長なコストとプロセスを回避するために、CSPは、複数の加入者アカウントを作成することなく、複数のユーザーペルソナを管理する手段をユーザーに提供すべきです(SHOULD)。これが不可能で、1 人の加入者に対して複数の加入者アカウントがサポートされている場合、CSP は、同じ加入者の不必要な再校正を回避する方法(たとえば、共通の識別子を介して、または生体認証または属性解決を通じてアカウントをリンクする)で、加入者アカウントを実装すべきです (SHOULD)。

    1 つのサブスクライバに対して複数のサブスクライバ アカウントが許可されている場合は、次の要件が適用されます。

    1. CSPは、詐欺の可能性を特定するために、複数のアカウントを持つ加入者をレビューおよび評価するためのプロセスを開発および文書化するものとします。
    2. CSP は、一意の政府識別子または共通のコア属性に関連付けられたすべてのアカウントのマッピングを維持するものとします (SHOULD)。
    3. CSP は、個人に、自分の ID に関連付けられたサブスクライバー アカウントの完全なリストを可視化するものとします。
    4. CSPは、加入者が自分の個人情報を使用した追加のアカウントの作成をブロックすることを許可すべきです(SHOULD)。

    RP が複数の CSP のサービスを使用する場合にも、同様のシナリオが発生する可能性があります。これはアカウントリンクと呼ばれ、[SP800-63C]のセクション3.8.1では、RP加入者アカウントと複数のCSP/IdPアカウントの関連付けについて説明しています。

    脅威とセキュリティに関する考慮事項

    このセクションは有益です。

    本人確認プロセスを効果的に保護するには、特定の申請者との取引全体を通じてセキュリティ制御とプロセスを階層化する必要があります。これを達成するには、脅威がどこでどのように発生し、登録を危険にさらす可能性があるかを理解する必要があります。ID 証明プロセスに対する脅威には、次の 4 つの一般的なカテゴリがあります。

    1. なりすまし: 攻撃者が別の正当な個人になりすまそうとする (個人情報の盗難など)
    2. 虚偽または詐欺的な表現: 攻撃者は、虚偽の ID の作成や ID に関する虚偽の主張を試みます (合成 ID 詐欺など)
    3. ソーシャルエンジニアリング攻撃: 攻撃者は、欺瞞や強制を使用して、結果として得られる加入者アカウントへのアクセスを制御することを最終目標として、ID 証明プロセス中に何らかの行動を起こすよう被害者を説得します
    4. インフラストラクチャ攻撃: 攻撃者は、インフラストラクチャ、データ、ソフトウェア、または CSP の ID 証明プロセスをサポートする人員の機密性、可用性、または整合性を侵害しようとします (分散型サービス拒否、内部脅威など)

    このセクションでは、なりすまし攻撃、虚偽または詐欺的な表現の脅威、ソーシャル エンジニアリング攻撃に焦点を当てます。インフラストラクチャの脅威は、従来のコンピューター セキュリティ制御 (侵入防御、記録保持、独立した監査など) によって対処され、このドキュメントの範囲外です。これらのガイドラインで提供されているもの以外の追加のセキュリティ制御については、[SP800-53]、連邦情報システムおよび組織に推奨されるセキュリティおよびプライバシー制御を参照してください。

    このセクションでは、個別の脅威タイプとしての AI に具体的に対処するガイダンスやコントロールは提供しません。代わりに、以下の緩和策とこれらのガイドラインの要件は、AI ツールを使用して攻撃者によって実行または拡張される可能性のある特定の脅威 (たとえば、AI が生成した偽造文書やメディアを使用して申請者になりすます) に対処します。

    隣接プログラムとの連携

    ID証明とCSP機能をサイバーセキュリティ、プライバシー、脅威インテリジェンス、およびプログラムの整合性チームと緊密に連携させることで、ID証明機能を継続的に改善しながら、ビジネス機能をより完全に保護できます。たとえば、プログラムの整合性チームによって収集された支払い詐欺データは、侵害された加入者アカウントや、ID 証明の実装における潜在的な弱点を示している可能性があります。同様に、脅威インテリジェンスチームは、ID 証明プロセスに影響を与える可能性のある新しい戦術、手法、手順の兆候を受け取る可能性があります。CSPとRPは、重要なセキュリティと不正の利害関係者の間で情報を交換するための一貫したメカニズムの確立に努める必要があります。CSP が RP の外部にある場合は、契約上および法的メカニズムを使用して、技術的および相互運用性に関する考慮事項など、これらのメカニズムを確立できます。収集、送信、または共有されるすべてのデータは最小限に抑えられ、詳細なプライバシーと法的評価の対象となる必要があります。

    収集とデータの最小化

    これらのガイドラインでは、要求された身元を検証し、要求された身元を申請者に関連付け、詐欺を軽減し、承認の決定を行うために使用できる属性を RP に提供するために必要な個人情報のみの収集と処理を許可します。身元証明サービスに不要な生体認証データを含む個人情報を処理すると、個人は自分のデータが期待や承認を超える方法で使用されているのではないかと懸念する可能性があります。これは、当惑、自律性の喪失、信頼の喪失などのプライバシーの問題につながる可能性があります。さらに、個人情報の保持は不正アクセスや不正使用に対して脆弱になる可能性があります。データの最小化により、不正アクセスや不正使用に対して脆弱な個人情報の量が減り、本人確認プロセスへの信頼が促進されます。

    社会保障番号

    これらのガイドラインにより、CSP は ID 解決で使用する属性として SSN を収集できます。ただし、SSN に過度に依存すると、誤用につながり、個人情報の盗難などの危害の危険にさらされる可能性があります。それにもかかわらず、SSN は CSP、特に SSN を使用して申請者を政府機関の記録に関連付ける連邦機関の身元解決を容易にする可能性があります。このドキュメントでは、SSN の役割を属性として認識し、その使用に適切な考慮を行っています。SSNの知識は、身元の証拠として機能するのに十分ではありません。

    可能であれば、CSP と政府機関は、ID 証明プロセス中に SSN の拡散と露出を制限するメカニズムを検討する必要があります。これは、属性検証プロセス中に SSN がサードパーティ プロバイダーに伝達される場合に特に重要です。可能な限り、個人の SSN がサードパーティのシステムによって公開、保存、または維持されるリスクを軽減するために、プライバシー保護技術とテクノロジーを適用する必要があります。この例としては、属性クレーム(バリデーターからのはい/いいえの応答など)を使用して、第三者が不必要に送信する必要を得ずにSSNの有効性を確認することが挙げられます。ID 証明プロセスのすべての属性と同様に、処理される各属性の価値とリスクはプライバシー リスク評価の対象となり、連邦政府機関は関連する PIA および SORN ドキュメントでそれに対処する場合があります。CSP は、申請者の SSN をコア属性と見なす場合、または ID 解決をサポートする場合、申請者の SSN を収集することが許可されます。

    通知と同意

    これらのガイドラインでは、CSP は、収集時に、ID 証明に必要な属性の収集と記録の維持の目的について、そのような属性が自発的か必須か、および属性を提供しなかった場合の結果を含め、申請者に明示的に通知することを義務付けています。さらに、第 3.1.11 条では、生体認証データを収集または処理する CSP に対し、このプロセスに関する詳細で公開されている情報を提供することが義務付けられています。効果的な通知では、ユーザー エクスペリエンス、設計基準、調査、およびコレクションから生じる可能性のあるプライバシー リスクの評価が考慮されます。応募者が属性が収集される理由を理解していると誤って推測したり、収集された情報が他のデータソースと組み合わされる可能性があると誤って推測したりするなど、さまざまな要因を考慮する必要があります。効果的な通知は、申請者が読んだり理解したりする可能性が低い、複雑で法律主義的なプライバシーポリシーや一般的な利用規約につながる単なる指針ではありません。

    同意により、個人は自分の情報の処理に関する決定に参加することができ、個人を特定できる情報の処理から生じるリスクの一部を組織から個人に移転します。少なくとも、これらのガイドラインでは、CSP は、第 3.1.11 条に規定されているように、生体認証データを収集して使用する前に、および本人確認セッションを記録する前に、ユーザーから同意を得る必要があります。RPは、CSPの選択に利用可能な選択肢、身分証明書の要件、関連するプライバシー通知、およびサービスにアクセスするための代替手段について、申請者に追加のガイダンスを提供する必要があります。

    使用制限

    これらのガイドラインでは、CSPは、処理属性から生じる可能性のあるプライバシーリスクに見合った予測可能性(つまり、情報システムによる個人情報の処理について個人、所有者、および事業者が信頼できる仮定を可能にする)と管理可能性(つまり、個人情報の変更、削除、選択的開示を含む個人情報のきめ細かな管理機能を提供する)を維持する対策を採用することを要求しています。本人確認、認証、承認、属性アサーション、関連する詐欺の軽減、または法律または法的手続きの遵守以外の目的。NIST プライバシー フレームワーク [NIST-Privacy] は、これらのリスクを管理し、プライバシー リスク管理の原則をサポートするためのフレームワークを提供します。

    CSP は、サブスクライバーへの非 ID サービスの提供など、属性を処理するためのさまざまなビジネス目的を持つ場合があります。ただし、サブジェクトに開示された目的以外の目的で属性を処理すると、追加のプライバシー リスクが生じる可能性があります。CSPは、追加の処理から生じるプライバシーリスクに見合った適切な対策を決定できます。たとえば、適用される法律、規制、またはポリシーがない場合、加入者が要求する非 ID サービスを提供するために属性を処理する際に同意を得る必要がない場合がありますが、通知は加入者が処理に関する信頼できる仮定 (つまり、予測可能性) を維持するのに役立つ場合があります。属性の他の処理には、同意の取得や、特定の属性の使用または開示 (つまり、管理性) に対する加入者へのより詳細な制御が必要となる、さまざまなプライバシー リスクが伴う場合があります。CSP は、これらの同意措置の受け入れを ID サービスを使用する条件にすることはできません。

    連邦政府機関は、提案された処理が許可された処理の範囲外であるかどうか、または適切なプライバシーリスク軽減策の範囲外であるかどうかについて質問がある場合は、SAOPに相談する必要があります。

    正す

    これらのガイドラインでは、CSPは、本人確認から生じる申請者の苦情や問題を是正するための効果的かつ安全なメカニズムを提供し、申請者がメカニズムを簡単に見つけてアクセスできるようにする必要があります。

    プライバシー法は、記録システムを維持する連邦CSPに対し、申請者が記録にアクセスして修正できるようにする手順に従うことを義務付けています。プライバシー法に関する声明には、申請者にアクセスまたは修正の要求を行う方法に関する指示を提供する、該当する SORN への言及 (第 3.3 条を参照) を含める必要があります。連邦政府以外の CSP は、情報源である第三者の連絡先情報など、同等の手順を用意する必要があります。

    申請者が身元を証明してオンライン登録プロセスを完了できない場合、CSPは、プロセスを完了するための代替方法の利用可能性を申請者に明確にする必要があります(たとえば、カスタマーサービスセンターで直接)。

    本人確認プロセスが成功しない場合、CSPは申請者に問題に対処するための手順を通知する必要がありますが、登録が失敗した理由の詳細を申請者に通知しないでください(たとえば、「あなたのSSNは当社が記録しているものと一致しませんでした」と申請者に通知しないでください)。

    プライバシーリスク評価

    これらのガイドラインでは、CSP がプライバシー リスク評価を実施する必要があります。プライバシーリスク評価を実施する際、CSPは以下を考慮する必要があります。

    1. その措置(追加の検証手順や記録の保持など)が、信頼の喪失など、申請者にプライバシーの問題を引き起こす可能性がある
    2. 問題が発生した場合の申請者への影響。CSPは、リスクの受け入れ、リスクの軽減、リスクの共有など、特定されたプライバシーリスクに対して取るあらゆる対応を正当化できる必要があります。申請者の同意はリスクを共有する一形態と見なされるため、申請者がこの共有リスクを評価し受け入れる能力を持っていることが合理的に期待できる場合にのみ使用する必要があります。

    政府機関固有のプライバシーコンプライアンス

    これらのガイドラインは、連邦政府の CSP に対する特定のコンプライアンス義務を対象としています。IDサービス開発の初期段階で政府機関のSAOPを関与させ、プライバシーリスクを評価および軽減し、ID証明を実施するための個人情報の収集が1974年のプライバシー法[プライバシー法]または2002年の電子政府法[E-Gov]をトリガーするかどうかなどのコンプライアンス要件について政府機関に助言することが重要ですPIAを実施するための要件。たとえば、本人確認に関しては、個人情報の収集と維持により、プライバシー法の要件がトリガーされ、本人確認の実施に必要な個人情報またはその他の属性の収集と維持により、新規または既存のプライバシー法 SORN による補償が必要になる可能性があります。

    SAOPは、PIAが必要かどうかを判断する際にも同様に支援できます。これらの考慮事項は、身元証明のみのためにプライバシー法SORNまたはPIAを開発するための要件として読まれるべきではありません。多くの場合、デジタル ID のライフサイクル全体を網羅する PIA と SORN を作成するか、政府機関がオンライン アクセスを確立しているプログラムや利点について議論する、より大規模なプログラムによる PIA の一部として ID 証明プロセスを含めることが最も理にかなっています。

    デジタル ID ライフサイクルには多くのコンポーネントがあるため、SAOP は個々のコンポーネントを認識し、理解することが重要です。たとえば、他のプライバシーアーティファクトは、身元証明サービス(データ使用契約、コンピューターマッチング契約など)を提供または使用する機関に適用できる場合があります。SAOPは、どのような追加要件が適用されるかを決定する際に政府機関を支援することができます。さらに、ID 証明の個々のコンポーネントを完全に理解することで、SAOP はコンプライアンスまたはその他の手段を通じてプライバシー リスクを徹底的に評価し、軽減できるようになります。

    ユーザーの観点から見ると、本人確認と登録の 3 つの主なステップは、1) 準備、2) ID 証明と登録、3) 登録後のアクションです。これらのステップは 1 つのセッションで発生する場合もあれば、各セッションの間にかなりの時間 (数日、数週間など) がある場合もあります。

    一般的なユーザビリティとステップ固有のユーザビリティに関する考慮事項は、以下のサブセクションで説明され、ユーザーの視点から説明されています。

    1973 年のリハビリテーション法第 508 条 [第 508 条] は、情報技術の障壁を取り除き、連邦機関に障害のある人々が電子技術や情報技術を利用できるようにすることを義務付けるために制定されました。これらのガイドラインは第 508 条の要件を直接主張するものではありませんが、ID サービス プロバイダーは第 508 条の規定に準拠することが期待されています。第 508 条への準拠を超えて、連邦政府機関とそのサービス プロバイダーは、一般に、ID システムのライフサイクル全体を通じてアクセシビリティが優先されるように、障害のある人々の経験を念頭に置いてサービスとシステムを設計することが期待されています。

    • ユーザーの不満を避けるために、本人確認と登録のプロセスを合理化し、各ステップをできるだけ明確かつ簡単にします。

    • 技術支援をどこでどのように受けられるかを明確に伝えます。たとえば、オンライン セルフサービス ポータルへのリンク、チャット セッション、ヘルプ デスク サポートの電話番号など、役立つ情報を提供します。理想的には、ユーザーが外部の介入なしに自分の登録準備の質問に答えることができるように、十分な情報を提供する必要があります。

    • どのような個人データが収集されているのか、またデータの収集が任意であるかどうかを明確に説明してください。さらに、データが共有される相手、保存場所、保護方法を示す情報を提供します。

    • 提示されたすべての情報が使用可能であることを確認します。
      • すべてのユーザー向け資料 (データ収集通知、入力可能なフォームなど) について、適切な情報設計慣行に従ってください。
      • 資料を平易な言葉で書き、専門用語は避けてください。必要に応じて、対象人口の識字率に合わせて言語を調整します。能動態と会話スタイルを使用します。要点を論理的に並べる。混乱を避けるために、同義語ではなく同じ単語を一貫して使用してください。読みやすさを高めるために、必要に応じて箇条書き、数字、書式設定を使用します。
      • フォント スタイル、サイズ、色、周囲の背景とのコントラストなど、テキストの読みやすさを考慮してください。最も高いコントラストは白地に黒です。ユーザーの視力レベルは異なるため、テキストの読みやすさは重要です。判読できないテキストは、ユーザーの理解ミスやユーザー入力エラーの原因となります(入力可能なフォームに記入するときなど)。電子資料にはサンセリフフォントスタイルを使用し、紙資料にはセリフフォントを使用します。可能であれば、混同しやすい文字を明確に区別しないフォント(文字「O」と数字「0」など)は避けてください。これは、確認コードの場合に特に重要です。テキストがディスプレイに収まる限り、最小フォント サイズは 12 ポイントです。
    • 代表的なユーザーと各ステップのユーザビリティ評価を行います。ユーザビリティ評価のための現実的な目標とタスク、および適切な使用コンテキストを確立します。

    準備

    ユーザーが ID 証明と登録プロセスに可能な限り準備できるようにすることは、ID 証明と登録の全体的な成功と使いやすさにとって重要です。

    このような準備は、ユーザーが必要な情報(必要な文書など)を使用可能な形式でタイムリーに受け取った場合にのみ可能です。これには、どのような身元証拠が必要になるかをユーザーに正確に認識させることが含まれます。ユーザーは、IALについて、または必要な身元証拠がFAIR、STRONG、またはSUPERIORと見なされるかどうかについて何も知る必要はありません。

    ユーザーが ID 証明と登録プロセスを続行するかどうか、およびセッションに何が必要かについて、十分な情報に基づいて決定できるようにするには、ユーザーに次の情報を提供します。

    • 各ステップで何が予想されるかなど、プロセス全体に関する情報
      • 予想される時間枠の明確な説明により、ユーザーはそれに応じて計画を立てることができます

    • ユーザーが価値提案を理解できるようにするための ID 証明の必要性と利点の説明

    • 目的の IAL の ID 証拠要件と、検証方法に関する情報を含む許容可能な証拠文書のリスト

    • 手数料がかかるかどうか、手数料がかかる場合は、許容される支払い方法の金額と種類に関する情報

    • 本人確認と登録のプロセスがオンサイトで実施されるか、リモートチャネル経由で実施されるか、またはその2つの組み合わせで実施されるか、およびユーザーが選択できるかどうかに関する関連情報
      • 場所に関する情報、ユーザーが希望する場所を選択できるかどうか、およびオンサイトの有人または無人セッションに必要なロジスティクス情報。ユーザーは、紛失や盗難のリスクが高まるため、特定の公共の場所 (スーパーマーケットと銀行など) に身元証拠を持ち込むことに消極的になる可能性があります。
      • リモートセッションの技術要件(インターネットアクセスの要件など)に関する情報
      • 待ち時間を最小限に抑えるために、リモート有人またはオンサイト有人の ID 証明セッションの予定を設定するオプション。ウォークインが許可されている場合は、予約なしで待ち時間が長くなる可能性があることをユーザーに明確に伝えてください。
        • 必要に応じて、本人確認の予約のスケジュール、リマインダーの取得、既存の予約のスケジュールの変更方法について明確な指示を提供します。
        • 予定のリマインダーを提供し、ユーザーが好みの予定のリマインダー方法 (郵便、ボイスメール、電子メール、テキスト メッセージなど) を指定できるようにします。ユーザーは、日付、時刻、場所、必要な身元証明の説明などの情報を必要とします。
    • 許可および必須の身元証拠と属性、各部分が任意か必須か、および完全な身元証拠セットを提供しなかった場合の結果に関する情報。ユーザーは、身元証拠に固有の要件 (出生証明書の印鑑など) を含む、身元証拠の特定の組み合わせを知る必要があります。これは、必要な身元証拠の入手が困難になる可能性があるため、特に重要です。
      • 可能であれば、必要な身元証拠を簡単に取得できるようにツールを実装します。
      • 未成年者または固有のニーズを持つ人に対する特別な要件をユーザーに通知します。たとえば、申請者の参照プロセスや信頼できる審査プロセスが利用可能かどうか、およびそれらのプロセスを使用するために必要な情報に関する情報をユーザーに提供します(セクション3.14を参照)。
      • フォームが必要な場合:
        • 入力可能なフォームを、ID 証明プロセスの前と最中に使用できるようにします。
        • ユーザーにプリンターへのアクセス権を要求しないでください。
        • フォームが長いとユーザーはイライラしやすく、エラーが発生しやすくなるため、ユーザーがフォームに入力しなければならない情報の量を最小限に抑えます。CSP は、スキャンされたドキュメントのデータや本人確認プロセス中に提供されたデータなど、認証されていないユーザーに個人情報が開示されない場合、フォーム フィールドを自動入力する場合があります。

    本人確認と登録

    次のユーザビリティに関する考慮事項は、ID の証明と登録に固有のものです。

    • ID 証明セッションの開始時に、手順をユーザーに通知します。準備ステップで説明したプロセスを覚えていることを期待しないでください。ID 証明セッションが準備手順の直後に続かない場合は、ID 証明と登録プロセスを完了するための一般的な時間枠をユーザーに明確に通知することが特に重要です。
    • 本人確認方法(リモートまたはオンサイトの無人など)に応じて、本人確認プロセスのステップバイステップのチュートリアルを含む別のビデオウィンドウを提供します。このようなチュートリアルや例が提供される場合、サービスプロバイダーは幅広いユーザーをカバーするための幅広いサポートオプションを用意する必要があります。ビデオ ウィンドウの代替手段には、口頭または書面による指示が含まれます。
    • 必要に応じて、ユーザーが本人確認の予定の時間またはタイプを再スケジュールするオプションを提供します。
    • 許可された必須の ID 証拠を含むチェックリストを提供して、ユーザーが ID 証明と登録プロセスを続行するために必要な ID 証拠を持っていることを確認します。ユーザーが完全な ID 証拠セットを持っていない場合は、部分的な ID 証明セッションを完了できるかどうか、または信頼できる審判を通じて例外処理を使用できるかどうか、または必要に応じて ID 証明例外処理の申請者参照を使用できるかどうかをユーザーに通知する必要があります。これは、ID証明の種類とデータ、サービス、および検証ソースへのアクセスが、IALのID証明要件を達成するために容易または容易に利用できない場合、国際的なユーザーにも当てはまります。信頼できる査読者と申請者参照は、例外処理を必要とするユーザーに、代替の ID 証明ワークフローとリスクベースの決定のための機能を提供することを目的としています。
    • 将来のフォローアップ セッションのために破棄または保持される情報、および将来のセッションを完了するために持参する必要がある身元証拠についてユーザーに通知します。理想的には、ユーザーは部分的な ID 証明セッションを完了するかどうかを選択できます。
    • その後のアクティビティ (証明の通知の配信など) の時間枠を含む、本人確認および登録プロセスの結果に関するユーザーの期待を設定します。
    • ユーザーが 1) 本人確認と登録プロセスが成功した直後にオーセンティケーターを受け取るか、2) オーセンティケーターを直接受け取るためのフォローアップの予約をスケジュールする必要があるか、または 3) オーセンティケーターを郵送で受け取り、受け取る場合はいつ受け取ることができるかを明確に示します。

    • サブスクライバ提供のオーセンティケータが許可されている場合は、サブスクライバ アカウントに登録する方法に関する情報をユーザに提供します。

    • ID 証明と登録のプロセス中に、CSP によって収集および処理されるデータなど、ユーザーに明示的な通知を提供するためのいくつかの要件があります。通知に関する詳細な要件については、第 3 条および第 7 条を参照してください。CSP は、ID 証明、認証、承認、または属性アサーション以外の目的で属性を使用することについてユーザーの同意を求めると、不快に感じる可能性があることに注意する必要があります。ユーザーが追加の収集または使用からどのような利益が得られるかを理解していない場合、同意を提供したり、プロセスを続行したりすることを望まない、または躊躇する可能性があります。したがって、CSP は、個人情報の追加処理によってどのようなメリットが得られるか、およびそのような処理に関連するリスクを軽減するために CSP が講じる手順について、ユーザーに徹底的な説明を提供することをお勧めします。さらに、CSP は、追加の処理をオプトアウトする機会をユーザーに提供する必要があります。

    • 確認コードは、申請者が将来の通信の目的で住所、電子メール アドレス、または電話番号にアクセスできることを確認するために使用されます。確認コードを使用する場合:
      • 確認コードの受信日、受信時期、コードの有効期間、到着方法(物理的な郵便、SMS、固定電話など)を事前にユーザーに通知します。
      • 確認コードがユーザーに配信されたら、ユーザーが登録しているサービスを思い出させ、コードの使用方法とコードの有効期間に関する指示を含めます。これは、セクション 3.8 で指定されている有効期間が短いことを考えると特に重要です。
      • QR コード (セクション 3.8 を参照) などの機械可読の光学ラベルを発行する場合は、QR コード スキャン機能 (許容される QR コード アプリケーションなど) を取得する方法に関する情報をユーザーに提供します。さらに、QR コードの代わりに人間が読めるコードを提供します。
      • 確認コードの有効期限が切れたり、紛失したりした場合の使用前に、ユーザーに通知します。
    • 申請者が 1 回のセッションで本人確認と登録を完了できない場合、または別の本人確認タイプ (リモート出席、オンサイト出席など) を介して本人確認を完了する必要がある場合は、継続コードが発行される場合があります。継続コードは、不完全な本人確認または登録プロセスへの申請者のリンクを再確立するために使用されます。継続コードが発行された場合:
      • 継続コード、使用理由、配信方法、ID 証明プロセスを完了するための使用方法に関する情報をユーザーに提供します。
      • 継続コードがユーザーに配信されたら、登録するサービスをユーザーに通知し、ID 証明または登録プロセスを完了するために何をする必要があるか、どこに行くべきかについての指示を含めます。
      • QR コード (第 3.9 節を参照) などの機械可読光学ラベルを発行する場合は、QR コード スキャン機能 (許容される QR コード アプリケーションなど) を取得する方法に関する情報をユーザーに提供します。さらに、QR コードの代わりに人間が読めるコードを提供します。
      • すべてのユーザーが必要なテクノロジーにアクセスできるわけではないため、不完全な ID 証明または登録プロセスへのリンクを再確立するための代替オプションをユーザーに提供します。
    • 本人確認と登録プロセスの最後に、次の操作を行います。
      • 登録が成功した場合は、本人確認と登録が成功したことを確認する証明の通知(セクション3.10を参照)と、実行する必要がある次のステップ(例:オーセンティケーターをいつ、どこで受け取るか、いつ郵送で到着するか)をサブスクライバーに送信します。
      • 登録が部分的に完了した場合 (たとえば、ユーザーが完全な ID 証拠セットを持っていない、ユーザーがプロセスの停止を選択した場合、またはセッション タイムアウトが原因の場合)、ユーザーに次のように伝えます。
        • どのような情報が破棄されるか。
        • 将来のフォローアップセッションのためにどのような情報が保持されるか。
        • 情報が保持される期間。そして
        • 今後のセッションに持参する必要がある身元証拠。
      • 登録が成功しない場合は、代替の本人確認と登録オプション(オンサイト出席、信頼できる審判の使用など)について明確な指示をユーザーに提供します。

    • ユーザーが登録セッション中にオーセンティケーターを受け取った場合は、その使用とメンテナンスに関する手順を提供します。たとえば、情報には、使用手順 (特に初回使用または初期化の要件が異なる場合)、オーセンティケーターの有効期限に関する情報、オーセンティケーターの保護方法、オーセンティケーターの紛失または盗難の場合の対処方法などが含まれます。

    • 対面とリモートの両方の ID 証明には、追加のユーザビリティに関する考慮事項が適用されます。
      • 登録セッションの開始時に、オペレーターまたはアテンダントは、自分の役割をユーザーに説明する必要があります (たとえば、オペレーターまたはアテンダントが登録セッションをユーザーに案内するか、黙って観察し、必要に応じてのみ対話するか)。
      • 登録セッションの開始時に、セッション中に退会してはならず、セッション全体を通してアクションを表示する必要があることをユーザーに通知します。
      • 登録セッション中に生体認証サンプルが収集されたら、キャプチャ プロセスを完了する方法について明確な指示をユーザーに提供します。指示は、プロセスの直前に行うのが最適です。ライブオペレーターからのガイダンスによる口頭での指示が最も効果的です(たとえば、生体認証キャプチャデバイスの場所と操作方法、キャプチャプロセスをいつ開始するか、生体認証キャプチャがいつ完了するかを知る方法をユーザーに指示します)。
    • リモート ID 証明はオンラインで行われるため、次のような一般的な Web ユーザビリティの原則に従ってください。
      • ユーザー インターフェイスを設計して、登録プロセスを順を追って説明します。
      • インターフェイスの簡素化や視覚補助の使用などの手法を通じて、ユーザーのメモリ負荷を軽減します。
      • インターフェイスの一貫性を持たせます。
      • 連続したステップに明確にラベルを付けます。
      • 出発点を明確にしてください。
      • 複数のプラットフォームとデバイスサイズをサポートします。
      • ナビゲーションの一線性を保ち、見つけやすく、わかりやすくします。

    登録後

    登録後とは、登録直後、オーセンティケーターを初めて使用する前の手順を指します。前述のように、ユーザーは、オーセンティケーターを受信または追加する予定の配信、集荷、または登録メカニズムについて、登録セッションの終了時に既に通知されています。

    登録後のユーザビリティに関する考慮事項には、次のものがあります。

    • ユーザーがオーセンティケーターが到着するまで待機する時間を最小限に抑えます。待ち時間が短くなると、ユーザーは情報システムやサービスにすばやくアクセスできるようになります。

    • 認証システムを受け取るために物理的な場所に行く必要があるかどうかをユーザーに通知します。以前に特定された予定とリマインダーのユーザビリティに関する考慮事項は引き続き適用されます。

    • オーセンティケーターとともに、特に初回使用または初期化に異なる要件がある場合、使用手順など、オーセンティケーターの使用と保守に関連する情報をユーザーに提供します。オーセンティケーターの有効期限に関する情報。認証システムを紛失または盗難した場合の対処方法。

    • ソーシャル エンジニアリングやフィッシング攻撃など、ID アカウントと関連する認証システムに対する一般的な脅威から身を守る方法に関する情報をユーザーに提供します。

    カスタマーサクセスに関する考慮事項

    顧客体験の主な側面は、ユーザー集団のニーズを予測し、その集団に適したソリューションを提供することです。これは、ID 解決、検証、検証のプロセスがそれぞれ独自のユーザー課題を導入するため、ID 証明のコンテキストではかなり困難になります。プルーフプロセス全体を分析し、一般的な課題を評価することで、CSPは「ユーザーがいる場所でユーザーに対応する」ソリューションを展開し、よりアクセスしやすく応答性の高いソリューションセットをユーザーに提供できます。このドキュメント内の規範的な推奨事項と一致して、次のサブセクションでは、ID 証明プロセス全体を通じてユーザーにオプションと選択肢を展開することの価値と、それらが軽減できるリスクについて説明します。

    複数のタイプのID証明をサポート

    単一のCSP、複数のCSP、またはRPカスタマーサービス担当者との取り決めによって展開されるかどうかにかかわらず、セクション2.1.3で定義されている複数のID証明タイプを使用すると、ユーザーはセキュリティを犠牲にしたり、例外処理に頼ったりすることなく、肯定的な証明結果を達成する機会が増えます。効果的な継続的改善メカニズムを使用してデプロイすると、複数のプルーフタイプを使用すると、次の一般的な問題の軽減をサポートできます。

    • 自動化されたプロセスやテクノロジーを多用するプロセスを完了できないユーザー(テクノロジーの流暢さが限られている、身体障害や認知障害など)、または特定のプロセス(自動生体認証など)の効果が低いユーザー
    • 観察やその他のセキュリティリスクが発生しやすい共有コンピューティングリソースに依存する可能性のあるユーザー
    • 高速インターネットや、リモートの本人確認プロセスを完了できるコンピューティングデバイスに確実にアクセスできないユーザー
    • リモートまたは自動化されたプロセスよりも人間の対話を好むユーザー
    • 自分の過失ではなく、リモートまたは自動アプローチに失敗した可能性のあるユーザー (例: 誤った拒否、キャプチャの失敗)

    セクション 4.1、4.2、および 4.3 では、各 IAL で使用できる ID 証明の許容されるタイプについて説明します。

    信頼できる複数の情報源と提携する

    信頼できる信頼できる情報源は、プルーフプロセス中に申請者が主張する身元データの正確性を確認する手段を提供します。これは、合成 ID 攻撃を防止し、RP がアクセスを許可し、ユーザーを既存のデータやアカウントに接続するための正確な情報を可能にするために不可欠なステップです。ただし、ソースは、特に多数のユーザーが単一のソースに依存している場合、不正確さ、問題、課題を引き起こす可能性もあります。このような場合、特定の種類のデータに関する問題が急増し、より多くのユーザー集団にまたがる可能性があります。たとえば、信頼できる情報源が単一のモバイル ネットワーク オペレーターからのデータを検証できないと、他のネットワーク オペレーターからのデータが利用可能で正確であっても、地域全体または経済グループに影響を与える可能性があります。

    したがって、ユーザーをより効果的にサポートするには、複数のデータソースの統合がCSPにとって重要です。このような統合は、CSPとRPが必要とするコア属性、ユーザーが利用できる証拠の種類、およびテストと継続的な評価プロセスに基づく特定のベンダーの有効性と母集団カバレッジに関する情報に基づいている必要があります。さらに、複数のベンダーを使用すると、フェイルオーバーまたはウォーターフォールプロセスが作成され、ID証明が正常に完了する可能性が高まり、ユーザーを例外処理プロセスにさらす必要性が制限されます。信頼できる信頼できる複数のソースを活用することで、堅牢なレコード セットを持っていないユーザーや、個々のソースで不完全なレコードを持っているユーザーのリスクを最小限に抑えることができます。

    セクション 2.4.2.3 およびセクション 2.4.2.4 は、データの検証と信頼できる信頼できる情報源の要件を扱います。

    堅牢で応答性の高い例外処理プロセスを提供します。

    本人確認プロセスにおけるエラーは、さまざまな理由で日常的に発生します。例外処理プロセスにより、ユーザーは、サブスクライバー アカウントの確立とサービスへのアクセスを正常に行いながら、ID 証明エクスペリエンスに関連する問題に対処できます。効果的な継続的改善メカニズムを使用して展開すると、例外処理プロセスは、次の一般的な問題を軽減するのに役立ちます。

    • 信頼できる情報源または権威ある情報源での履歴が限られているユーザー
    • 信頼できる情報源または権威のある情報源を持つ記録が不完全、不正確、または過去の個人情報の盗難に起因するエラーの影響を受ける可能性があるユーザー
    • 一般的な形式の身元証拠を持たないユーザー、または証拠が紛失、破壊、盗難、またはその他の方法で侵害されたユーザー
    • インターネット帯域幅の制限、リモートでの本人確認プロセスを完了できるデバイスへのアクセスの制限、技術的な流暢さの制限、身体的または認知的障害、または特定のプロセス(自動生体認証など)の効果が低いために、自動化されたプロセスまたはテクノロジーを大量に消費するプロセスを完了できないユーザー
    • 属性が変更され、レコードにそれらの変更が反映されていないユーザー
    • 名前の書式設定やデータ入力設計の結果、レコードを ID システムと照合するのが難しい可能性があるユーザー

    セクション3.14は、例外処理プロセスの特定の要件を提供します。例外処理プロセスでは、ある程度のリスクが受け入れられるため、多くの場合、他のコントロールと組み合わせて RP の露出を制限できます。これには、承認、資格、アクセスの制限や、例外処理プロセスで確立されたサブスクライバーアカウントに対する追加の監視の適用が含まれます。RP がこれらのコントロールを実装するには、CSP は、例外処理プロセスがアサーション、API 経由、または別のリアルタイム メカニズムを使用して使用されたことを示すインジケーターを RP に提供する必要があります。

    用語集

    このセクションは有益です。

    デジタル ID の分野では、さまざまな用語が使用されています。多くの定義は以前のバージョンの SP 800-63 と一致していますが、このリビジョンでは一部が変更されています。これらの用語の多くは、単一の一貫した定義を欠いているため、ここでの用語の定義方法に細心の注意を払う必要があります。

    出願者
    本人確認と登録のプロセスを経ている被験者。
    応募者紹介
    申請者の身元、申請者に関連する特定の属性、または個人の状況に関連する条件(緊急事態、ホームレスなど)を保証できる申請者の代表者。
    承認された暗号化
    暗号化アルゴリズム、ハッシュ関数、ランダム ビット ジェネレーター、または連邦情報処理標準 (FIPS) が承認または NIST が推奨する同様の技術。承認されたアルゴリズムと手法は、FIPS または NIST の推奨事項で指定または採用されています。
    主張
    IdP から RP へのステートメントで、加入者の認証イベントに関する情報が含まれています。アサーションには、サブスクライバーの ID 属性を属性値、派生属性値、および属性バンドルの形式で含めることもできます。
    属性
    誰かまたは何かに起因する品質または特性。ID 属性は、加入者の ID (名前、生年月日、住所など) に関する属性です。
    属性の検証
    属性のセットが正確で、現実の ID に関連付けられていることを確認するプロセスまたは行為。検証を参照してください。
    認証
    認証を参照してください。
    認証された保護チャネル
    接続イニシエーター (クライアント) が受信者 (サーバー) を認証した承認された暗号化を使用する暗号化通信チャネル。認証された保護されたチャネルは、機密性とアクティブな仲介者に対する保護を提供するために暗号化され、ユーザー認証プロセスで頻繁に使用されます。トランスポート層セキュリティ (TLS) とデータグラム トランスポート層セキュリティ (DTLS) [RFC9325] は、受信者によって提示された証明書がイニシエータによって検証される認証された保護チャネルの例です。特に指定がない限り、認証された保護チャネルでは、サーバーがクライアントを認証する必要はありません。サーバーの認証は、多くの場合、各サーバーで個別にではなく、信頼できるルートにつながる証明書チェーンを介して行われます。
    認証
    請求者が、加入者アカウントにバインドされた 1 つ以上の認証者の所有と制御を証明し、そのアカウントに関連付けられた加入者であることを証明するプロセス。
    認証保証レベル(AAL)
    認証プロセスの強度を記述するカテゴリ。
    認証
    加入者が所有および制御するもの(暗号化モジュールやパスワードなど)で、請求者の身元を認証するために使用されるもの。「認証システムの種類」と「多要素認証システム」を参照してください。
    信頼性
    データがソースとされるものから発生したプロパティ。
    権威ある情報源
    発行元からの正確な情報にアクセスできる、または発行元からの正確な情報のコピーを検証したエンティティで、CSP は、ソースが ID 証明中に申請者によって提供された ID 属性または証拠の有効性を確認できると高い信頼度を持っています。発行元は、信頼できる情報源である場合もあります。多くの場合、信頼できるソースは、ID 証明の検証フェーズで使用する前に、代理店または CSP のポリシー決定によって決定されます。
    許可する
    アクセスを許可する決定で、通常はサブジェクトの属性を評価することで自動化されます。
    生体認証参照
    パスポートにデジタルで保存された顔画像、国民IDカードの指紋細部テンプレート、話者認識のためのガウス混合モデルなど、個人に帰属し、データベース内の生体認証比較の対象として使用される1つ以上の保存された生体認証サンプル、テンプレート、またはモデル。
    生体認証サンプル
    指紋画像を含むレコードなど、生体認証特徴抽出前の生体認証特性のアナログまたはデジタル表現。
    バイオメトリクス
    生物学的または行動的特徴に基づいて個人を自動的に認識します。生物学的特徴には、指紋、掌紋、顔の特徴、虹彩と網膜のパターン、声紋、静脈パターンが含まれますが、これらに限定されません。行動特性には、キーストロークのケイデンス、スマートフォンの持ち方、画面の圧力、タイピング速度、マウスや携帯電話の動き、ジャイロスコープの位置などが含まれます。
    要求者
    1 つ以上の認証プロトコルを使用して ID を検証するサブジェクト。
    要求された ID
    未検証および未検証の個人的属性に関する申請者の宣言。
    コア属性
    CSP が ID 証明とサービスの提供に必要であると判断し、文書化した一連の ID 属性。
    資格情報サービス プロバイダー (CSP)
    ID サービスへの ID 証明申請者やサブスクライバ アカウントへの Authenticator の登録などの機能を持つ信頼できるエンティティ。CSP は、独立したサード パーティである場合があります。
    信頼できる情報源
    ID 証拠と属性情報の正確性を提供または検証できるエンティティ。信頼できるソースは、ID 証明プロセスを通じて検証された属性情報、または信頼できるソースまで追跡できる属性情報にアクセスできるか、複数のソースから取得した ID 属性情報を保持し、正確性、一貫性、および最新性についてデータの相関性がチェックされます。
    デジタルアイデンティティ
    特定のコンテキスト内のサブジェクトを一意に記述する属性または属性のセット。
    デジタル署名
    秘密キーを使用してデータにデジタル署名し、公開キーを使用して署名を検証する非対称キー操作。デジタル署名は、信頼性保護、整合性保護、否認防止のサポートを提供しますが、機密性やリプレイ攻撃の保護は提供しません。
    非解離性
    システムの動作要件を超えて、個人やデバイスに関連付けることなく、個人情報やイベントを処理すること。[NISTIR8062]
    在籍
    CSP/IdP が、ID 証明に成功した申請者に加入者アカウントを提供し、オーセンティケーターをバインドして永続的なアクセスを許可するプロセス。
    エントロピー
    攻撃者がシークレットの価値を判断するために直面する不確実性の量。エントロピーは通常ビットで表されます。nビットのエントロピーを持つ値は、一様に分布したnビットランダム値と同じ程度の不確実性を持ちます。
    連邦情報処理標準 (FIPS)
    米国商務省の一部であるNISTによって開発された、連邦省庁による採用および使用に関する標準。FIPS は、共通のレベルの品質、セキュリティ、相互運用性を達成するために、情報技術のトピックに対処します。FIPS ドキュメントは、FIPS ホームページ https://www.nist.gov/itl/fips.cfm からオンラインで入手できます。
    フェデレーション
    一連のネットワークシステム間で ID および認証情報を伝達できるプロセス。
    フェデレーション保証レベル (FAL)
    認証イベントとサブスクライバ属性を RP に伝達するためにフェデレーショントランザクションで使用されるプロセスを記述するカテゴリ。
    識別子
    特定のコンテキスト内の単一の一意のエンティティ (個人、デバイス、セッションなど) に関連付けられ、そのコンテキスト内の他のエンティティに割り当てられないデータ オブジェクト。
    同一性
    「デジタル ID」を参照してください。
    ID 保証レベル (IAL)
    主体が主張するアイデンティティが自分の本当のアイデンティティであるという確信度を伝えるカテゴリー。
    \clearpage
    身元証拠
    要求された ID の現実世界の存在をサポートする情報またはドキュメント。身元証拠は、物理的 (運転免許証など) またはデジタル (モバイル運転免許証やデジタル アサーションなど) の場合があります。証拠は、検証(つまり、信憑性と正確性の確認)と検証(つまり、申請者が証拠の真の所有者であることを確認する)の両方をサポートする必要があります。
    本人確認
    サブジェクトの主張された ID の保証を確立するために、サブジェクトに関する情報を収集、検証、検証するために使用されるプロセス。
    ID プロバイダー (IdP)
    サブスクライバのアサーションを作成し、そのアサーションを RP に送信するフェデレーション トランザクションのパーティ。
    ID 解決
    CSPがサービスを提供する母集団のコンテキスト内で個人を一意に区別するために、申請者に関する情報を収集するプロセス。
    本人確認
    検証を参照してください。
    インジェクション攻撃
    攻撃者が信頼できない生体認証情報またはメディアをプログラムまたはプロセスに提供する攻撃。たとえば、これには、身元証拠の偽造画像、ユーザーの偽造ビデオ、または証拠検証テクノロジーを無効にするためのモーフィング画像の挿入、またはユーザー検証のための生体認証と視覚的比較が含まれます。
    発行元(発行元)
    身元証拠として使用できるデータ、デジタル証拠 (つまり、アサーション)、または物理的な文書の生成を担当する機関。
    知識ベース検証(KBV)
    申請者の主張された身元を確認する目的で、個人に関連する個人情報または個人情報の知識を検証するプロセス。KBV には、身元解決を目的とした個人属性の収集は含まれません。
    管理
    個人情報の改ざん、削除、選択的開示など、きめ細かな管理を行う機能の提供[NISTIR8062]
    自然人
    合成でも人工でもない現実の人間。
    ネットワーク
    請求者と他の当事者の間でメッセージを転送するために使用されるオープン通信媒体(通常はインターネット)。特に明記されていない限り、ネットワークはオープンであり、当事者間の任意の時点でアクティブ(なりすまし、セッションハイジャックなど)および受動的(盗聴など)攻撃を受けると想定されます(例:請求者、検証者、CSP、RP)。
    否認防止
    特定の取引を実行したことを誤って否定する個人から保護する機能。
    1対1(1:1)の比較
    個人からの生体認証サンプルを生体認証参照と比較して、比較スコアを生成するプロセス。
    オンライン攻撃
    攻撃者が正規の検証者を持つ要求者の役割を引き受けるか、認証チャネルをアクティブに変更する認証プロトコルに対する攻撃。
    オンラインサービス
    ネットワーク (通常はインターネット) を介してリモートでアクセスされるサービス。
    個人情報
    単独で、または特定の個人にリンクまたはリンク可能な他の情報と組み合わせて、個人の身元を識別または追跡するために使用できる情報。
    実践ステートメント
    認証プロセスにおける当事者(CSPや検証者など)が従う慣行の正式な声明。通常、当事者の方針と慣行を説明し、法的拘束力を持つ場合があります。
    予測
    個人、所有者、および事業者が、個人情報および情報システムによるその処理について信頼できる仮定を可能にします。[NISTIR8062]
    プレゼンテーション攻撃
    生体認証システムの動作を妨害することを目的として、生体認証データキャプチャサブシステムへのプレゼンテーション。
    プレゼンテーション攻撃検出 (PAD)
    プレゼンテーション攻撃の自動判定プレゼンテーション攻撃の決定方法(すなわち、生体検出)のサブセットには、解剖学的特徴または自発的または非随意的な反応の測定と分析が含まれ、キャプチャの時点で存在する生体被験者から生体認証サンプルがキャプチャされているかどうかを判断します。
    プライバシー影響評価(PIA)
    個人情報がどのように収集、使用、共有、維持されているかを分析する方法。PIA は、プログラムまたはシステムの開発ライフサイクル全体を通じてプライバシー リスクを特定し、軽減するために使用されます。また、情報の取り扱いがプライバシーに関する法律、規制、およびポリシー要件に準拠していることを確認するのにも役立ちます。
    秘密鍵
    エンティティに一意に関連付けられ、公開されない公開キー暗号化アルゴリズムで使用される暗号化キー。非対称鍵 (公開鍵) 暗号システムでは、秘密鍵には対応する公開鍵があります。アルゴリズムによっては、秘密鍵を使用して次のことを行うことができます。

    1. 対応する公開鍵を計算し、

    2. 対応する公開鍵によって検証できるデジタル署名を計算し、

    3. 対応する公開鍵で暗号化された鍵を復号化する、または

    4. キーアグリーメントトランザクション中に共有シークレットを計算します。

    プロセスアシスタント
    校正プロセスのサポートを提供するが、意思決定やリスクベースの評価 (翻訳、文字起こし、アクセシビリティのサポートなど) をサポートしない個人。
    加工
    個人情報の収集、保持、ログ記録、生成、変換、使用、開示、転送、または廃棄を含むがこれらに限定されない、個人情報に対して実行される操作または一連の操作。[NISTIR8062]
    プルーフ剤
    身元証明セッションに参加するように訓練され、身元証拠を物理的に検査したり、申請者を身元証拠と比較したりするなど、限定的なリスクベースの決定を下すことができるCSPのエージェント。
    雅号
    正式名称以外の名称。
    偽名
    主題を識別するための仮名の使用。
    仮名識別子
    RP が加入者に関する何も推測できないが、RP が複数の対話を 1 つの加入者に関連付けることを許可する、無意味だが一意の識別子。
    公開キー
    エンティティに一意に関連付けられ、公開できる公開キー暗号化アルゴリズムで使用される暗号化キー。非対称鍵 (公開鍵) 暗号システムでは、公開鍵には対応する秘密鍵があります。公開鍵は誰でも知っている可能性があり、アルゴリズムによっては、次の目的で使用できます。

    1. 対応する秘密鍵を使用して生成されたデジタル署名を検証します。

    2. 対応する秘密鍵を使用して復号化できる鍵を暗号化する、または

    3. キーアグリーメントトランザクション中に共有シークレットを計算します。

    公開鍵証明書
    認証局の秘密鍵によって発行され、デジタル署名されたデジタル文書で、識別子を加入者の公開鍵にバインドします。証明書は、証明書で識別されたサブスクライバが秘密鍵を単独で制御し、アクセスできることを示します。[RFC5280] も参照してください。
    公開鍵インフラストラクチャ (PKI)
    証明書と公開鍵と秘密鍵のペアを管理するために使用される一連のポリシー、プロセス、サーバー・プラットフォーム、ソフトウェアおよびワークステーション (公開鍵証明書の発行、保守、および失効機能を含む)。
    登録
    登録を参照してください。
    証明書利用者 (RP)
    通常、トランザクションを処理したり、情報やシステムへのアクセスを許可したりするために、加入者の ID に対する検証者のアサーションに依存するエンティティ。
    遠隔
    対面ではなく、ネットワークを介して接続されたデバイスを介して実行されるプロセスまたはトランザクション。
    解決
    ID 解決を参照してください。
    リスク評価
    システムの運用から生じる組織の運営(ミッション、機能、イメージ、評判)、組織資産、個人、その他の組織に対するリスクを特定、推定、優先順位付けするプロセス。リスク評価はリスク管理の一部であり、脅威と脆弱性の分析を組み込み、計画または実施されているセキュリティ制御によって提供される軽減策を考慮します。これは「リスク分析」と同義です。
    リスク管理
    組織の業務(ミッション、機能、イメージ、レピュテーションなど)、組織資産、個人、その他の組織に対する情報セキュリティリスクを管理するプログラムおよびサポートプロセスで、(i)リスク関連活動のコンテキストの確立、(ii)リスクの評価、(iii)リスクの決定後への対応、および(iv)リスクの長期にわたる監視が含まれます。
    RPサブスクライバーアカウント
    IdP からのサブスクライバ アカウントの RP のビューに基づいて、フェデレーション システムで RP によって確立および管理されるアカウント。RP サブスクライバ アカウントは 1 つ以上のフェデレーション識別子に関連付けられており、サブスクライバは IdP とのフェデレーション トランザクションを通じてアカウントにアクセスできます。
    プライバシー担当上級機関(SAOP)
    政府機関がプライバシー要件を遵守し、プライバシーリスクを管理し、個人情報を含むすべての政府機関の行動とポリシーがプライバシーに与える影響を考慮する責任者。
    セッション
    加入者とエンドポイント(RP または CSP)の間の永続的な相互作用。セッションは認証イベントで始まり、セッション終了イベントで終わります。セッションは、加入者のソフトウェア(ブラウザ、アプリケーション、OS など)が RP に提示して、セッションと認証イベントの関連性を証明できるセッション シークレットを使用してバインドされます。
    ソーシャルエンジニアリング
    個人をだまして機密情報を漏らしたり、不正アクセスを行わせたり、個人と関係を築いて詐欺を犯したりして自信や信頼を得る行為。
    件名
    個人、組織、デバイス、ハードウェア、ネットワーク、ソフトウェア、またはサービス。これらのガイドラインでは、被験者は自然人です。
    加入者
    CSP ID サービスに登録されている個人。
    サブスクライバーアカウント
    ID サービスに登録されている各サブスクライバーに対して CSP によって確立されたアカウントで、サブスクライバーに関する情報と、サブスクライバーに登録されているオーセンティケーターのレコードが含まれています。
    補足的なコントロール
    これらのガイドラインの保証レベルで指定されているコントロールに加えて、特定の脅威または攻撃に対処するために追加できるコントロール。
    合成ID詐欺
    個人情報の組み合わせを使用して、個人または団体を捏造して、個人的または金銭的利益のために不正な行為を行うこと。
    記録システム (SOR)
    個人に関する情報を含み、機関の管理下にあるレコードのコレクション。レコードは、個人の名前、識別番号、記号、またはその他の識別子によって取得できます。
    記録システム通知 (SORN)
    連邦機関が記録システムを説明するために連邦官報に発行する通知。
    トランザクション
    デジタルトランザクションを参照してください。
    信託契約
    CSP、IdP、および RP がフェデレーション トランザクションに参加して、加入者と RP 間の認証セッションを確立できる一連の条件。
    トラストアンカー
    別の信頼できるエンティティ(公開鍵証明書など)によって保証されているためではなく、ハードウェアまたはソフトウェアに直接組み込まれているか、帯域外の手段を介して安全にプロビジョニングされているために信頼される公開鍵または対称鍵。トラスト アンカーには、その範囲を制限する名前またはポリシーの制約がある場合があります。
    信頼できる審判員
    申請者が定義された IAL 証明プロセスの予想される要件を満たすことができない場合に、申請者の ID 証明ケースに関してリスクベースの決定を下すように訓練された CSP のエージェント。
    ユーザビリティ
    特定のユーザーが、特定の使用コンテキストで有効性、効率性、満足度を持って特定の目標を達成するために製品を使用できる程度。[ISO/IEC9241-11]
    検証
    申請者から提供された証拠と属性が本物で正確であり、現実の身元に関連付けられていることをチェックおよび確認するプロセスまたは行為。属性の検証を参照してください。
    検証
    身元証明を受ける申請者が、検証された身元属性および関連する証拠によって表される主張された実生活の身元を保持していることを確認するプロセスまたは行為。本人確認の代名詞。
    ベリファイア
    認証プロトコルを使用して、請求者が 1 つ以上の認証子を所有および制御していることを検証することにより、請求者の ID を確認するエンティティ。これを行うには、検証者はオーセンティケーターと加入者アカウントのバインドを確認し、加入者アカウントがアクティブであることを確認する必要があります。

    変更ログ

    この付録は有益です。

    この付録では、最初のリリース以降にSP 800-63Aに加えられた変更の概要を示します。

    • グローバル要件と IAL 固有の要件を分離して、保証レベルに関係なく ID サービスの設計を容易にします
    • セクションを再編成して、特定の要件を提供する前に、一般的な ID 証明要件を導入します
    • コア属性の概念を導入します
    • ID 属性の収集を ID 証拠の収集から切り離します
    • 不正管理のガイダンスと要件を導入
    • デジタルインジェクション防止と偽造メディア検出に関するガイダンスと要件を提供します
    • CSP 固有のプライバシー リスク評価の要件と、結果を政府機関の評価プロセスに統合するための考慮事項を追加します。
    • 信頼できる情報源を含むように、許容可能な証拠と属性検証のソースを拡張します
    • 信頼できる査読者や申請者の参照の使用に関する要件など、例外処理の概念と要件を紹介します
    • 更新されたIAL1を通じて、リスクの低いアプリケーションの要件を提供
    • IAL 1 と 2 の証拠収集要件を調整します。
    • IAL 1 および 2 での本人確認のための非生体認証オプションを提供
    • サブスクライバー アカウントに対する新しいガイダンスと要件を追加
